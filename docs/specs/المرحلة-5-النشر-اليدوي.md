# المرحلة ٥ — النشر اليدوي المُيسّر (Spec)

> بعد اعتماد العميل للبوست، يأتي آخر متر: إخراجه للمنصة. في هذه النسخة النشر **يدوي مُيسّر** — لا API نشر تلقائي. مهمّة المنصّة أن تجعل الخطوة اليدوية بضغطتين: نصّ منسّق جاهز للّصق + صورة جاهزة للتنزيل + زر يفتح composer المنصة، مع تذكير عند الموعد حتى لا يُنسى البوست، ثم تعليمه "منشوراً" يدوياً.
>
> يبني على نموذج Prisma القائم (`Post`, `ImageAsset`) ويطبّق حدود المنصات من [15-مواصفات-المنصات.md](../../15-مواصفات-المنصات.md). يستهلك مخرجات [16-معمارية-المحرّك.md](../../16-معمارية-المحرّك.md) (`Post` بحالة `approved`).
>
> NestJS · multi-tenant منطقي (`tenantId`). أسماء الأنواع والمسارات بالإنجليزية (code-facing)؛ الشرح بالعربي.
>
> يغطّي: **FR-11** (نسخ/تنزيل/فتح)، **FR-12** (تذكير داخل التطبيق + بريد)، **FR-13** (تعليم منشوراً). القصص: US-5.1، US-5.2، US-5.3.

## المبادئ

- **يدوي مُيسّر لا مؤتمت:** لا اتصال بـ APIs النشر للمنصات (تكلفة موافقات/مخاطر تعليق حساب). دور المنصّة تقليل الاحتكاك اليدوي إلى أدنى حد، لا إزالته.
- **النص هو مصدر الحقيقة، الـdeep link مساعد:** المنصّات لا تسمح غالباً بتعبئة composer مسبقاً عبر URL. لذا التدفّق الواقعي = نسخ النص للحافظة ثم فتح composer فارغ ولصق. الـ`deepLink` يفتح المكان الصحيح فقط، لا يحمل النص.
- **التنسيق يطبّق حدود المنصة:** التصدير يعيد فرض حدود ملف ١٥ (طول النص، عدد الهاشتاقات، موضع الرابط) — البوست المعتمد يُفترض ملتزماً، لكن التصدير يتحقّق ولا يثق.
- **التذكير قناة قابلة للتوسعة:** بنية `NotificationChannel` مجرّدة؛ الإطلاق يشحن `in_app` + `email`، وإضافة `whatsapp` لاحقاً = قناة جديدة بلا إعادة تصميم للجدولة أو التدفّق.
- **عزل tenant في كل مسار:** كل عملية (`export`, `mark-published`, `reminders`) مقيّدة بـ`tenantId` للطالب؛ لا وصول عابر للـtenants.
- **انتقالات حالة صريحة:** `mark-published` انتقال أحادي الاتجاه `approved → published` فقط؛ أي حالة أخرى تُرفض بوضوح.
- **التذكير لا يُسقطه فشل قناة:** فشل قناة واحدة (بريد مثلاً) لا يمنع الأخرى (in_app)؛ كل قناة تُسلَّم وتُسجَّل مستقلة.

## المدخلات والأنواع (Data Types)

```ts
// مُعاد من ملف 16 للوضوح
type Platform = 'linkedin' | 'x'
type PostStatus = 'draft' | 'pending_review' | 'approved' | 'published'

// حمولة التصدير — كل ما يحتاجه العميل للنشر اليدوي بضغطة
interface ExportPayload {
  postId: string
  platform: Platform
  formattedText: string     // نص جاهز للّصق: المتن + الهاشتاقات بترتيب المنصة (الرابط مُدار حسب platform)
  imageUrl?: string         // رابط تنزيل الصورة (ImageAsset.url) إن وُجدت
  deepLink: string          // رابط فتح composer/المنصة (web intent / share URL حيث يتاح)
  link?: ExportLink         // الرابط الخارجي إن وُجد، وأين يوضع
  charCount: number         // العدّ الموزون الفعلي (twitter-text لـX) للعرض/التحقّق
  limitMax: number          // الحد الأقصى للمنصة من platform-limits
  notes: string[]           // إرشادات يدوية للعميل (مثل: "ضع الرابط في أول ردّ" لـX)
}

interface ExportLink {
  url: string
  placement: 'in_body' | 'first_reply'   // X → first_reply؛ LinkedIn → in_body بلا بطاقة معاينة
}

// تذكير مجدول لبوست
type NotificationChannelId = 'in_app' | 'email'   // 'whatsapp' لاحقاً (V2)

interface Reminder {
  id: string
  tenantId: string
  postId: string
  channel: NotificationChannelId
  remindAt: string          // ISO؛ افتراضياً = Post.scheduledAt
  status: ReminderStatus
  createdAt: string
}

type ReminderStatus = 'scheduled' | 'sent' | 'failed' | 'cancelled'

// إشعار داخل التطبيق — سجل إشعارات الـtenant (يكتبه InAppChannel)
type NotificationType = 'reminder'   // أنواع أخرى لاحقاً

interface Notification {
  id: string
  tenantId: string
  userId?: string           // المستخدم المستهدف داخل الـtenant إن وُجد؛ غيابه = إشعار على مستوى الـtenant
  type: NotificationType
  title: string
  body: string
  postId?: string           // البوست المرتبط إن وُجد (لفتح المحرّر/التصدير)
  readAt?: string           // ISO؛ غيابه = غير مقروء
  createdAt: string
}

// نتيجة تعليم البوست منشوراً
interface MarkPublishedResult {
  postId: string
  status: 'published'
  publishedAt: string
}
```

## المكوّنات وواجهات REST

> كل المسارات تحت حماية الـAuth + حقن `tenantId`. الاستجابات مغلّفة بمظروف الـAPI القياسي للمشروع (مُلخّص هنا بالحمولة فقط).

### `PublishingController`

#### `GET /posts/:id/export?platform=linkedin|x`
يبني `ExportPayload` لبوست **معتمد** (FR-11 / US-5.1, US-5.2).

- يقرأ `Post` (+`ImageAsset` عبر `postId` الفريد) ضمن `tenantId` الطالب.
- يبني `formattedText` عبر `ExportFormatter` (يطبّق حدود ملف ١٥).
- يبني `deepLink` عبر `DeepLinkBuilder` للمنصة المطلوبة.
- يرجع `ExportPayload`. الصورة **لا** تُجمَّع في الحمولة؛ يُعاد `imageUrl` فقط ويتولّى العميل التنزيل (زر منفصل).
- `platform` الافتراضي = `Post.platform`؛ المُعامل يتيح تصدير نفس النص لمنصة بديلة عند الحاجة.

#### `POST /posts/:id/mark-published`
ينقل الحالة `approved → published` يدوياً (FR-13 / US-5.3).

```ts
// Request body
interface MarkPublishedRequest {
  publishedAt?: string   // ISO؛ افتراضياً now()
}
// Response: MarkPublishedResult
```
- يتحقّق أن الحالة الحالية = `approved` فقط؛ غير ذلك → `409 invalid_status_transition`.
- يضبط `status='published'` و`publishedAt`، ويُلغِ أي `Reminder` مجدول مُعلّق لنفس البوست (`scheduled → cancelled`) — لا داعي لتذكير بعد النشر.

#### `POST /reminders`
يجدول تذكيراً لبوست عند موعد النشر (FR-12 / US-5.2).

```ts
// Request body
interface CreateReminderRequest {
  postId: string
  channels?: NotificationChannelId[]   // افتراضياً ['in_app', 'email']
  remindAt?: string                    // ISO؛ افتراضياً Post.scheduledAt
}
// Response: Reminder[]   (تذكير لكل قناة مطلوبة)
```
- يتطلّب `remindAt` صريحاً أو `Post.scheduledAt` موجوداً؛ غياب الاثنين → `422 remind_at_required`.
- يُنشئ سجل `Reminder` لكل قناة ويجدول **BullMQ delayed job** بتأخير = `remindAt - now()`.
- `remindAt` في الماضي → `422 remind_at_in_past`.

#### `DELETE /reminders/:id`
يلغي تذكيراً مجدولاً (`scheduled → cancelled`) ويزيل الـjob من الطابور. تذكير `sent` لا يُلغى (`409`).

#### `GET /posts/:id/reminders`
يسرد تذكيرات البوست (للعرض في الواجهة)، مقيّدة بـ`tenantId`.

### `NotificationController`

#### `GET /notifications`
يسرد إشعارات الـtenant (التي كتبها `InAppChannel`)، الأحدث أولاً.

```ts
// Query params (اختيارية)
interface ListNotificationsQuery {
  unreadOnly?: boolean   // true → غير المقروءة فقط (readAt = null)
}
// Response: Notification[]
```
- مقيّد بـ`tenantId` الطالب؛ لا يسرّب إشعارات tenant آخر.
- إن وُجد `userId` على الإشعار، يُقصر السرد على إشعارات المستخدم الطالب + الإشعارات على مستوى الـtenant (`userId = null`).

#### `PATCH /notifications/:id/read`
يعلّم إشعاراً مقروءاً (يضبط `readAt = now()`).

```ts
// Response: Notification   (بعد التحديث)
```
- مقيّد بـ`tenantId`؛ إشعار خارج الـtenant → `404 not_found`.
- مثاليّ (idempotent): إشعار مقروء مسبقاً يبقى بنفس `readAt` ويرجع `200`.

### `ExportFormatter` (خدمة)
يبني `formattedText` مطبّقاً حدود ملف ١٥ عبر `platform-limits` config (لا قيم منثورة):

- **LinkedIn:** المتن ضمن ٣٬٠٠٠ حرف، الخطّاف في أول ~١٤٠ حرف، ٣–٥ هاشتاقات في النهاية، الرابط **داخل المتن** مع ملاحظة "احذف بطاقة المعاينة" في `notes`. `placement='in_body'`.
- **X:** العدّ الموزون عبر `twitter-text` (لا `.length`)، ضمن ٢٨٠، ١–٢ هاشتاق، الرابط **في ردّ منفصل** (`placement='first_reply'`) مع ملاحظة في `notes`.
- يحسب `charCount` و`limitMax`؛ تجاوز الحد → يُرجع البوست لمراجعة (لا يُصدَّر بوست مكسور) عبر `422 exceeds_platform_limit`.

### `DeepLinkBuilder` (خدمة)
يبني `deepLink` لفتح composer/المنصة. **قيد واقعي:** المنصّتان لا تدعمان غالباً تعبئة النص مسبقاً عبر URL لمنشور عادي، لذا الرابط يفتح composer فارغ أو share intent ويُعتمد على لصق النص من الحافظة:

| المنصة | الـdeep link | الملاحظة |
|---|---|---|
| LinkedIn | web: `https://www.linkedin.com/feed/?shareActive=true` (يفتح composer)؛ app intent على الموبايل حيث يتاح | لا تعبئة نص مسبقة موثوقة → الاعتماد على لصق `formattedText` |
| X | web intent: `https://x.com/intent/post?text=<urlencoded>` (يدعم نصاً مسبقاً ضمن حدّ URL)؛ app `twitter://post` على الموبايل | الـintent يقبل نصاً قصيراً؛ النص الطويل يُلصق يدوياً — `notes` توضّح ذلك |

- يحقن النص في X intent عند ملاءمته للطول؛ خلاف ذلك يفتح composer فارغ.
- يكتشف web/app حسب سياق العميل (الواجهة تمرّر مؤشّر منصة العميل؛ خارج نطاق هذا الـspec تفاصيل اكتشاف الجهاز — يكفي إرجاع رابط web آمن افتراضياً).

### `ReminderScheduler` (BullMQ)
- عند `POST /reminders`: ينشئ delayed job لكل قناة بـ`delay = remindAt - now()`، يحمل `{ reminderId, postId, tenantId, channel }`.
- عند النضوج: الـworker يجلب البوست **جاهزاً** (مع `ExportPayload` مبني)، ثم يفوّض للقناة عبر `NotificationDispatcher`.
- يضبط `Reminder.status` إلى `sent` أو `failed` حسب نتيجة القناة.
- جدولة آمنة لتعدّد العمّال (idempotent عبر `reminderId`؛ تكرار التسليم محمي بفحص `status != 'sent'`).

### `NotificationDispatcher` + `NotificationChannel` (بنية قابلة للتوسعة)

```ts
interface NotificationChannel {
  id: NotificationChannelId
  send(payload: ReminderNotification): Promise<DeliveryResult>
}
interface ReminderNotification {
  tenantId: string
  postId: string
  export: ExportPayload     // البوست جاهزاً داخل التذكير (FR-12: "والبوست جاهز قدّامك")
  remindAt: string
}
interface DeliveryResult { delivered: boolean; error?: string }
```
- `InAppChannel`: يكتب صفّاً في جدول `Notification` (سجل إشعارات الـtenant) بـ`type='reminder'`, `tenantId`, `postId`, و`title`/`body` يحملان ملخّص البوست + إشارة لفتح المحرّر/التصدير. يُقرأ لاحقاً عبر `GET /notifications`.
- `EmailChannel`: يرسل بريداً (قالب RTL عربي) بالنص الجاهز + رابط الصورة + زر فتح المنصة.
- **التوسعة:** `WhatsappChannel` لاحقاً (V2 / fast-follow) = تنفيذ جديد للـinterface يُسجَّل في الـDispatcher؛ لا تغيير في الجدولة أو المسارات. إضافة `'whatsapp'` لـ`NotificationChannelId` فقط.

## التدفّقات

### التدفّق ١ — تصدير ونشر يدوي (US-5.1)
```
العميل يفتح بوستاً معتمداً
  → GET /posts/:id/export?platform=...
  → ExportFormatter يبني formattedText (حدود ملف 15) + DeepLinkBuilder يبني deepLink
  → الواجهة تعرض: زر "نسخ النص" (→ الحافظة) · زر "تنزيل الصورة" (imageUrl) · زر "افتح المنصة" (deepLink)
  → العميل ينسخ، يفتح composer، يلصق، ينشر يدوياً
```

### التدفّق ٢ — تذكير مجدول (US-5.2 / FR-12)
```
عند اعتماد بوست له scheduledAt (أو طلب صريح)
  → POST /reminders { postId, channels:['in_app','email'] }
  → Reminder لكل قناة + BullMQ delayed job (delay = remindAt - now)
  ⏰ عند النضوج:
  → worker يبني ExportPayload (البوست جاهز) → NotificationDispatcher
  → InAppChannel.send (يكتب صفّ Notification) + EmailChannel.send (مستقلّتان)
  → كل Reminder.status ← sent | failed
  → العميل يتلقّى التذكير والبوست جاهز قدّامه + زر فتح المنصة
```

### التدفّق ٣ — تعليم منشوراً (US-5.3 / FR-13)
```
العميل ينشر يدوياً ثم يرجع للمنصّة
  → POST /posts/:id/mark-published
  → فحص الحالة = approved (وإلا 409)
  → status ← published, publishedAt ← now
  → إلغاء أي Reminder مجدول مُعلّق لنفس البوست (scheduled → cancelled)
```

## معالجة الأخطاء

| المسار | الحالة | الكود | السلوك |
|---|---|---|---|
| export | بوست ليس `approved` | `409 not_approved` | لا تصدير قبل الاعتماد |
| export | تجاوز حدّ المنصة بعد التنسيق | `422 exceeds_platform_limit` | يُعاد للمراجعة بدل تصدير مكسور |
| export | لا صورة (`ImageAsset` غير موجود) | `200` | يُصدَّر بنص فقط؛ `imageUrl` غائب (متّسق مع overlay-fail في ملف ١٦) |
| mark-published | الحالة ليست `approved` | `409 invalid_status_transition` | الانتقال أحادي `approved → published` فقط |
| reminders | لا `remindAt` ولا `scheduledAt` | `422 remind_at_required` | لا جدولة بلا موعد |
| reminders | `remindAt` في الماضي | `422 remind_at_in_past` | لا تذكير لموعد فائت |
| reminder job | فشل قناة واحدة | — | تُسجَّل `failed` للقناة؛ القنوات الأخرى تكمل (لا إسقاط متبادل) |
| reminder job | البوست حُذف/تغيّر قبل النضوج | — | يُلغى التذكير بهدوء (`cancelled`)؛ لا تذكير ليتيم |
| notifications | إشعار خارج `tenantId` الطالب (read) | `404 not_found` | عزل tenant (لا تسريب وجود) |
| أي مسار | بوست خارج `tenantId` الطالب | `404 not_found` | عزل tenant (لا تسريب وجود) |

## الاعتمادات

- **ملف ١٦ (المحرّك):** يستهلك `Post` بحالة `approved` ومخرجاته (`text`, `hashtags`, `ImageAsset`).
- **ملف ١٥ (مواصفات المنصات):** `platform-limits` config + `twitter-text` للعدّ الموزون — مصدر حدود التنسيق في `ExportFormatter`.
- **BullMQ + Redis:** نفس بنية الطابور المستخدمة في "خطة الشهر" (ملف ١٦) — delayed jobs للتذكير.
- **قناة البريد:** مزوّد بريد المشروع (transactional) بقالب RTL عربي.
- **نموذج Prisma:** يبني على `Post` و`ImageAsset` القائمين، ويضيف نموذجَي `Reminder` و`Notification` الجديدين. جداول هذه المرحلة (`Reminder`, `Notification`) تُضاف بـ**migration مستقلّة في هذه المرحلة** (لا تُعدَّل migrations سابقة — LR-004).

```prisma
// migration هذه المرحلة (جديدة، مستقلّة)
model Notification {
  id        String    @id @default(cuid())
  tenantId  String
  userId    String?
  type      String    // NotificationType: 'reminder'
  title     String
  body      String
  postId    String?
  readAt    DateTime?
  createdAt DateTime  @default(now())

  @@index([tenantId])
}
```

## خارج النطاق

- **النشر التلقائي / API المنصات:** أي اتصال بـ LinkedIn/X APIs للنشر المُعبّأ = V2.
- **قناة واتساب:** مؤجّلة (fast-follow / V2) — البنية جاهزة لها بلا إعادة تصميم.
- **الواجهة (frontend):** أزرار النسخ/التنزيل/الفتح، عرض الإشعارات، شاشات التذكير = المرحلة ٧ (UI).
- **إنشاء/توليد البوست:** المرحلة ١ والمحرّك (ملف ١٦).
- **اكتشاف الجهاز التفصيلي (app vs web deep link):** يكفي إرجاع web link آمن؛ منطق الكشف الدقيق في الواجهة.
- **تنفيذ migration نموذجَي `Reminder` و`Notification`:** أرض الـbackend (الـspec يحدّد الشكل والحقول؛ الكتابة الفعلية للـmigration في الـbackend، migration مستقلّة لهذه المرحلة — LR-004).

## معايير القبول

- **AC-1 (FR-11/US-5.1):** لبوست `approved`، `GET /posts/:id/export` يرجع `formattedText` ملتزماً بحدود المنصة (طول، هاشتاقات، موضع الرابط) و`imageUrl` (إن وُجد) و`deepLink` صالحاً.
- **AC-2 (US-5.1):** التصدير يفصل النص عن الصورة — `formattedText` للحافظة و`imageUrl` لزر تنزيل منفصل.
- **AC-3 (FR-11):** `deepLink` يفتح composer/المنصة الصحيحة؛ ولـX يحقن النص في web intent عند ملاءمة الطول، مع `notes` توضّح حدود التعبئة المسبقة.
- **AC-4 (FR-12/US-5.2):** `POST /reminders` يجدول تذكيراً عبر `in_app` + `email` افتراضياً، يُطلَق عند `remindAt` (= `scheduledAt` افتراضياً) والبوست جاهز داخل التذكير.
- **AC-5 (FR-12):** فشل قناة واحدة لا يمنع الأخرى؛ كل قناة تُسلَّم وتُسجَّل حالتها مستقلة.
- **AC-6 (التوسعة):** إضافة قناة واتساب تتم بتنفيذ `NotificationChannel` جديد وتسجيله، بلا تعديل في `ReminderScheduler` أو المسارات.
- **AC-7 (FR-13/US-5.3):** `POST /posts/:id/mark-published` ينقل `approved → published` فقط؛ أي حالة أخرى ترجع `409`، ويُلغي التذكيرات المُعلّقة للبوست.
- **AC-8 (عزل):** كل المسارات تحترم `tenantId`؛ بوست tenant آخر يرجع `404`.
- **AC-9 (الحدود):** التنسيق يقرأ القيم من `platform-limits` config (لا منثورة) ويستخدم `twitter-text` للعدّ في X؛ تجاوز الحد يرجع `422` بدل تصدير مكسور.
