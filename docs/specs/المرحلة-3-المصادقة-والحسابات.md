# المرحلة 3 — المصادقة + الـTenant + إدارة الحسابات

> **المشروع:** أثر (منصّة SaaS عربية لإدارة المحتوى عبر المنصّات)
> **الـstack:** NestJS (backend) + Prisma + PostgreSQL
> **المرجع:** PRD — `FR-14`، قصص المستخدم `US-1.1`
> **حالة الوثيقة:** Spec تصميمي (design spec) — لا يتضمّن خطوات تنفيذ TDD.

---

## 1. نظرة عامة (Overview)

تغطّي هذه المرحلة الطبقة الأساسية للهوية والعزل في منصّة أثر:

1. **التسجيل والدخول** عبر البريد الإلكتروني وكلمة المرور، مع إنشاء `Tenant` و`User` وبدء **تجربة مجانية مدّتها 7 أيام** تلقائياً عند التسجيل.
2. **عزل الـTenant المنطقي** (logical multi-tenancy): كل سجلّ في قاعدة البيانات يحمل `tenantId`، وكل طلب لا يصل إلا إلى بيانات الـTenant صاحب الجلسة. لا توجد قاعدة بيانات منفصلة لكل عميل (no per-customer DB).
3. **إدارة ملفات الحسابات** (`AccountProfile`) لتخصيص المحتوى لكل منصّة (LinkedIn / X) — **بلا ربط API فعلي** مع تلك المنصّات في هذه المرحلة؛ الملفات وصفية فقط (handle, platform).
4. مسار `GET /me` لاسترجاع هوية المستخدم الحالي وسياق الـTenant والاشتراك.
5. **حقوق PDPL (NFR-5):** مساري `POST /me/export` (تصدير بيانات الـTenant) و`DELETE /me` (حذف/تعطيل الحساب مع موافقة) — تحقيق حقّ الحذف/التصدير في هذه المرحلة.

المخرج النهائي: واجهات REST آمنة، آلية JWT (access + refresh)، و`TenantContext` يُحقن في كل طلب ويمنع تسريب النطاق (scope leakage).

---

## 2. المبادئ (Principles)

- **العزل أولاً (isolation-first):** لا يُكتب أي استعلام يلمس بيانات قابلة للعزل دون شرط `tenantId`. الـTenant يُؤخذ من الـJWT المُتحقَّق منه، **لا** من جسم الطلب أو الـquery params (منعاً لتزوير النطاق).
- **أقل صلاحية ممكنة (least privilege):** الـtoken يحمل فقط ما يلزم (`sub`, `tenantId`). لا أدوار موسّعة في هذه المرحلة.
- **الكود والمعرّفات بالإنجليزية فقط:** أسماء الأنواع والواجهات والمسارات والحقول كلها إنجليزية. الشرح فقط بالعربية.
- **البناء على النموذج الموجود:** نوسّع مخطط Prisma الحالي (`Tenant`, `User`, `AccountProfile`, `Subscription`) دون كسر الحقول القائمة.
- **فصل المخاوف (separation of concerns):** `AuthModule` للمصادقة، `TenantModule` للعزل، `AccountProfileModule` لإدارة الحسابات، `UserModule` لـ`/me`.
- **التوافق مع PDPL:** تشفير كلمة المرور، **وتنفيذ حقّ الحذف والتصدير (NFR-5) في هذه المرحلة** عبر مساري `POST /me/export` و`DELETE /me` (انظر 4.2).
- **الأخطاء بصيغة موحّدة:** كل خطأ يتبع مظروف أخطاء ثابت (`error envelope`) لتسهيل استهلاك الواجهة.

## 3. أنواع البيانات (Data Types)

### 3.1 نموذج البيانات (Prisma — توسعة على الموجود)

نبني على النموذج القائم. الحقول الجديدة في هذه المرحلة موسومة بـ`// new`. أما حقول `Subscription` (`trialEndsAt`, `currentPeriodEnd`, `cancelAtPeriodEnd`) و enum `SubscriptionStatus` فهي **معرّفة مسبقاً في خطة الأساس (Sprint 0)** وتظهر هنا للسياق فقط — هذه المرحلة لا تنشئها، بل تنشئ فقط صفّ `Subscription` بحالة `trialing` عند التسجيل (دون migration جديد للحقول). كما أن العلاقة `Tenant.subscriptions Subscription[]` علاقة **واحد-لمتعدّد (تاريخية)**، لذا لا يوجد `@unique` على `Subscription.tenantId` (دورة `past_due ↔ active` تتطلّب تاريخاً متعدّداً).

```prisma
model Tenant {
  id            String         @id @default(cuid())
  name          String
  users         User[]
  subscriptions Subscription[]   // one-to-many (historical) — defined in Sprint 0 base plan
  accounts      AccountProfile[]
  createdAt     DateTime       @default(now())   // new
}

model User {
  id           String   @id @default(cuid())
  tenantId     String
  tenant       Tenant   @relation(fields: [tenantId], references: [id])
  email        String   @unique
  name         String?
  passwordHash String                              // new — never expose
  createdAt    DateTime @default(now())            // new

  @@index([tenantId])
}

model AccountProfile {
  id             String   @id @default(cuid())
  tenantId       String                            // new — required for isolation
  tenant         Tenant   @relation(fields: [tenantId], references: [id])
  brandProfileId String
  platform       Platform                          // 'linkedin' | 'x'
  handle         String?
  createdAt      DateTime @default(now())          // new

  @@index([tenantId])
}

// Subscription is one-to-many on Tenant (historical): no @unique on tenantId,
// so a tenant can accumulate multiple rows across the past_due <-> active cycle.
// trialEndsAt, currentPeriodEnd, cancelAtPeriodEnd and the SubscriptionStatus
// enum are ALL defined in the Sprint 0 base plan — shown here as existing context,
// not introduced by this phase.
model Subscription {
  id                 String             @id @default(cuid())
  tenantId           String             // no @unique — one-to-many (historical)
  tenant             Tenant             @relation(fields: [tenantId], references: [id])
  status             SubscriptionStatus // 'trialing' | 'active' | 'past_due' | 'canceled'
  plan               String
  trialEndsAt        DateTime?          // existing (Sprint 0 base plan)
  currentPeriodEnd   DateTime?          // existing (Sprint 0 base plan)
  cancelAtPeriodEnd  Boolean            @default(false) // existing (Sprint 0 base plan)
  createdAt          DateTime           @default(now())  // existing (Sprint 0 base plan)

  @@index([tenantId])
}

enum Platform {
  linkedin
  x
}

// SubscriptionStatus — defined in Sprint 0 base plan (shown for context).
enum SubscriptionStatus {
  trialing
  active
  past_due
  canceled
}
```

> **ملاحظة عزل:** إضافة `tenantId` إلى `AccountProfile` ضرورية. بدونها يصبح العزل معتمداً على سلسلة علاقات (`brandProfile -> tenant`) قابلة للخطأ والتسريب. القاعدة: **كل جدول قابل للعزل يحمل `tenantId` مباشرةً.**

### 3.2 أنواع TypeScript / DTOs

```ts
// JWT payload — السطر الوحيد لمصدر الحقيقة في العزل
interface JwtPayload {
  sub: string;        // userId
  tenantId: string;   // tenant scope — مصدر العزل الوحيد
  type: 'access' | 'refresh';
  iat: number;
  exp: number;
}

interface AuthTokens {
  accessToken: string;   // قصير العمر (~15m)
  refreshToken: string;  // طويل العمر (~7d)
  tokenType: 'Bearer';
  expiresIn: number;     // ثوانٍ لعمر الـaccessToken
}

interface RegisterDto {
  tenantName: string;    // اسم المؤسسة/المساحة
  email: string;         // @IsEmail
  password: string;      // @MinLength(8) — يُتحقَّق من قوّتها
  name?: string;
}

interface LoginDto {
  email: string;         // @IsEmail
  password: string;      // @IsNotEmpty
}

interface RefreshDto {
  refreshToken: string;  // @IsJWT
}

// السياق المُحقَن في كل طلب مُصادَق
interface TenantContext {
  userId: string;
  tenantId: string;
}

// AccountProfile DTOs
interface CreateAccountProfileDto {
  brandProfileId: string;
  platform: 'linkedin' | 'x';
  handle?: string;
}

interface UpdateAccountProfileDto {
  handle?: string;       // PATCH جزئي — platform غير قابل للتعديل بعد الإنشاء
}

// مظروف الاستجابة العام للأخطاء
interface ErrorEnvelope {
  statusCode: number;
  error: string;         // رمز ثابت: EMAIL_ALREADY_EXISTS, INVALID_CREDENTIALS, ...
  message: string;       // رسالة قابلة للعرض
}
```

## 4. المكوّنات والواجهات (Components & Interfaces)

### 4.1 الوحدات (NestJS Modules)

| الوحدة | المسؤولية |
|---|---|
| `AuthModule` | التسجيل، الدخول، تجديد الـtoken، التحقّق من الـJWT، تشفير كلمة المرور. |
| `TenantModule` | `TenantGuard` + `@CurrentTenant()` decorator + استخراج `TenantContext`. |
| `UserModule` | `GET /me` وقراءة بيانات المستخدم. |
| `AccountProfileModule` | CRUD لـ`AccountProfile` ضمن نطاق الـTenant. |
| `PrismaModule` | عميل Prisma المشترك (+ امتداد عزل اختياري، انظر 6.3). |

### 4.2 مسارات REST (REST Endpoints)

كل المسارات تحت البادئة العامة `/api/v1` المعرّفة في خطة الأساس عبر `app.setGlobalPrefix('api/v1')` (ليست قراراً خاصاً بهذه المرحلة) — المسارات أدناه نسبية لها.

#### المصادقة (Auth)

| الطريقة | المسار | مُصادَق؟ | الوصف |
|---|---|---|---|
| `POST` | `/auth/register` | لا | ينشئ `Tenant` + `User` + `Subscription(trialing, 7d)` ويعيد `AuthTokens`. |
| `POST` | `/auth/login` | لا | يتحقّق من البريد + كلمة المرور ويعيد `AuthTokens`. |
| `POST` | `/auth/refresh` | لا (refresh token في الجسم) | يتحقّق من الـrefresh token ويصدر زوج tokens جديداً. |

```
POST /auth/register
Body: RegisterDto
201 → AuthTokens
409 → ErrorEnvelope { error: 'EMAIL_ALREADY_EXISTS' }
422 → ErrorEnvelope (validation)

POST /auth/login
Body: LoginDto
200 → AuthTokens
401 → ErrorEnvelope { error: 'INVALID_CREDENTIALS' }

POST /auth/refresh
Body: RefreshDto
200 → AuthTokens
401 → ErrorEnvelope { error: 'INVALID_REFRESH_TOKEN' | 'TOKEN_EXPIRED' }
```

#### المستخدم الحالي (Me)

| الطريقة | المسار | مُصادَق؟ | الوصف |
|---|---|---|---|
| `GET` | `/me` | نعم (access) | يعيد هوية المستخدم + الـTenant + ملخّص الاشتراك. |

```
GET /me
Headers: Authorization: Bearer <accessToken>
200 → {
  user:   { id, email, name },
  tenant: { id, name },
  subscription: { status, plan, trialEndsAt }
}
401 → ErrorEnvelope { error: 'UNAUTHENTICATED' | 'TOKEN_EXPIRED' }
```

#### حقوق PDPL — التصدير والحذف (Data Rights — NFR-5)

تحقّق هذه المرحلة **NFR-5 (حقّ الحذف/التصدير)** الذي كان مؤجّلاً سابقاً. المساران مُصادَقان، ويعملان حصراً على نطاق الـTenant المستخرج من الـJWT.

| الطريقة | المسار | مُصادَق؟ | الوصف |
|---|---|---|---|
| `POST` | `/me/export` | نعم (access) | يصدّر كل بيانات الـTenant الحالي: المستخدمون، `BrandProfile`، `Posts` (إضافةً إلى `AccountProfile` والاشتراكات) كـJSON قابل للتنزيل. |
| `DELETE` | `/me` | نعم (access) | يحذف/يعطّل حساب الـTenant وبياناته مع موافقة صريحة: `soft-delete` فوري ثم `purge` مجدول. |

```
POST /me/export
Headers: Authorization: Bearer <accessToken>
200 → {
  exportedAt: string,            // ISO timestamp
  tenant: { id, name },
  users:    User[],              // بدون passwordHash
  brandProfiles: BrandProfile[],
  posts:    Post[],
  accountProfiles: AccountProfile[],
  subscriptions:   Subscription[]
}
401 → ErrorEnvelope { error: 'UNAUTHENTICATED' | 'TOKEN_EXPIRED' }

DELETE /me
Headers: Authorization: Bearer <accessToken>
Body: { confirm: true }          // موافقة صريحة إلزامية
202 → { status: 'scheduled_for_deletion', purgeAfter: string }  // ISO timestamp
422 → ErrorEnvelope { error: 'CONFIRMATION_REQUIRED' }          // confirm != true
401 → ErrorEnvelope { error: 'UNAUTHENTICATED' | 'TOKEN_EXPIRED' }
```

> **منطق `POST /me/export`:** يجمع كل السجلّات المرتبطة بـ`tenantId` الحالي في حِزمة JSON واحدة (المستخدمون مع استبعاد `passwordHash`، `BrandProfile`، `Posts`، `AccountProfile`، الاشتراكات). كل استعلام مُقيّد بـ`where: { tenantId: ctx.tenantId }` — لا تسريب نطاق.
>
> **منطق `DELETE /me`:** يتطلّب `confirm: true` في الجسم (موافقة صريحة) وإلا `422 CONFIRMATION_REQUIRED`. عند التأكيد: `soft-delete` فوري (وسم `deletedAt` على الـTenant وإبطال الجلسات/التوكنات) يحجب الوصول مباشرةً، ثم **purge مجدول** (مهمة خلفية بعد فترة احتجاز) يمحو فعلياً كل سجلّات الـTenant. يُرجِع `202 Accepted` مع `purgeAfter`. (تفاصيل المهمة المجدولة وفترة الاحتجاز تُسلَّم لـbackend/devops.)

#### ملفات الحسابات (Account Profiles — CRUD)

كل المسارات مُصادَقة، ومُقيّدة تلقائياً بنطاق الـTenant.

| الطريقة | المسار | الوصف |
|---|---|---|
| `GET` | `/accounts` | يسرد كل ملفات حسابات الـTenant الحالي فقط. |
| `POST` | `/accounts` | ينشئ ملف حساب جديد (`tenantId` يُحقن من السياق، لا من الجسم). |
| `PATCH` | `/accounts/:id` | يحدّث `handle` لملف ضمن نطاق الـTenant. |
| `DELETE` | `/accounts/:id` | يحذف ملفاً ضمن نطاق الـTenant. |

```
GET /accounts          200 → AccountProfile[]
POST /accounts         Body: CreateAccountProfileDto   201 → AccountProfile
PATCH /accounts/:id    Body: UpdateAccountProfileDto    200 → AccountProfile
DELETE /accounts/:id   204
404 → ErrorEnvelope { error: 'ACCOUNT_NOT_FOUND' }   // أيضاً عند وجوده لكن خارج النطاق
```

> **قاعدة العزل في الـCRUD:** أي عملية على `/accounts/:id` تُنفَّذ كـ`where: { id, tenantId }`. سجلّ موجود لكن يخصّ tenant آخر يُعامَل كـ`404` (لا `403`) لعدم تسريب وجوده.

### 4.3 الحرّاس والمزخرِفات (Guards & Decorators)

```ts
// يتحقّق من access token ويُحقن TenantContext في request
@Injectable()
class JwtAuthGuard implements CanActivate { /* verify access JWT */ }

// يضمن وجود tenantId صالح في السياق ويمنع الطلبات بلا نطاق
@Injectable()
class TenantGuard implements CanActivate { /* assert ctx.tenantId */ }

// param decorator: يستخرج TenantContext من request (لا يقرأ من الجسم أبداً)
const CurrentTenant = createParamDecorator(/* req.tenantContext */);

// مثال استخدام في كنترولر
@UseGuards(JwtAuthGuard, TenantGuard)
@Get('/accounts')
list(@CurrentTenant() ctx: TenantContext) {
  return this.accounts.listForTenant(ctx.tenantId);
}
```

## 5. التدفّقات (Flows)

### 5.1 تدفّق التسجيل (Register)

1. التحقّق من `RegisterDto` (صحّة البريد، طول كلمة المرور ≥ 8).
2. فحص تكرار البريد: `SELECT user WHERE email = ?`. إن وُجد → `409 EMAIL_ALREADY_EXISTS` مع رسالة ترشد للدخول (`US-1.1`).
3. تشفير كلمة المرور (`argon2id` مفضّل، أو `bcrypt cost=12`).
4. **معاملة ذرّية (transaction)** تنشئ معاً:
   - `Tenant { name: tenantName }`
   - `User { tenantId, email, name, passwordHash }`
   - `Subscription { tenantId, status: 'trialing', plan: 'trial', trialEndsAt: now + 7d }`
5. إصدار `AuthTokens` (access + refresh) بـ`payload = { sub: user.id, tenantId }`.
6. الرد `201` مع `AuthTokens`.

> أي فشل في الخطوة 4 يُرجِع المعاملة كاملةً (لا tenant يتيم بلا اشتراك أو مستخدم).

### 5.2 تدفّق الدخول (Login)

1. التحقّق من `LoginDto`.
2. جلب المستخدم بالبريد. إن لم يوجد → `401 INVALID_CREDENTIALS` (نفس الرسالة لعدم كشف وجود البريد).
3. مقارنة كلمة المرور بالـhash. فشل → `401 INVALID_CREDENTIALS`.
4. إصدار `AuthTokens` والرد `200`.

### 5.3 تدفّق التجديد (Refresh)

1. التحقّق من توقيع الـrefresh token وصلاحيته ونوعه (`type === 'refresh'`).
2. فشل التحقّق → `401 INVALID_REFRESH_TOKEN`؛ منتهٍ → `401 TOKEN_EXPIRED`.
3. إصدار زوج جديد (access + refresh) — **rotation**: الـrefresh الجديد يحلّ محلّ القديم.

### 5.4 تدفّق طلب مُصادَق (Authenticated Request)

1. `JwtAuthGuard` يستخرج `Authorization: Bearer` ويتحقّق من الـaccess token.
2. فشل/انتهاء → `401`.
3. يُبنى `TenantContext { userId: sub, tenantId }` ويُحقن في `request`.
4. `TenantGuard` يؤكّد وجود `tenantId`.
5. طبقة الخدمة تستخدم `ctx.tenantId` في كل استعلام — **لا تقرأ النطاق من الجسم**.

## 6. معالجة الأخطاء (Error Handling)

### 6.1 جدول الأخطاء

| الحالة | الرمز | `error` | الرسالة (للعرض) |
|---|---|---|---|
| بريد مكرّر عند التسجيل | `409` | `EMAIL_ALREADY_EXISTS` | "هذا البريد مسجّل مسبقاً. سجّل الدخول بدلاً من ذلك." |
| بيانات دخول خاطئة | `401` | `INVALID_CREDENTIALS` | "البريد أو كلمة المرور غير صحيحة." |
| access token منتهٍ | `401` | `TOKEN_EXPIRED` | "انتهت الجلسة، جدّد الدخول." |
| refresh token غير صالح | `401` | `INVALID_REFRESH_TOKEN` | "جلسة غير صالحة، سجّل الدخول من جديد." |
| طلب بلا مصادقة | `401` | `UNAUTHENTICATED` | "يلزم تسجيل الدخول." |
| فشل تحقّق المدخلات | `422` | `VALIDATION_ERROR` | تفاصيل الحقول. |
| مورد خارج النطاق أو غير موجود | `404` | `ACCOUNT_NOT_FOUND` | "العنصر غير موجود." |
| حذف الحساب بلا تأكيد صريح | `422` | `CONFIRMATION_REQUIRED` | "يلزم تأكيد الحذف صراحةً." |

### 6.2 مبادئ المعالجة

- **عدم كشف الوجود:** أخطاء الدخول موحّدة، وموارد tenant آخر تُرجِع `404` لا `403`.
- **`GlobalExceptionFilter`** يحوّل كل الاستثناءات إلى `ErrorEnvelope` ثابت.
- **عدم تسريب الـhash:** `passwordHash` لا يخرج أبداً في أي استجابة (يُستبعد عبر serializer/select صريح).

### 6.3 منع تسريب النطاق (Scope Leakage Prevention)

تطبيقاً لدرس مستفاد من الدراسة (عزل الصلاحيات):

- `tenantId` **يأتي حصراً** من الـJWT المُتحقَّق منه؛ تجاهل أي `tenantId` في الجسم أو الـquery.
- يُوصى بامتداد Prisma (Client Extension) يحقن `tenantId` تلقائياً في كل `where` و`create` للجداول القابلة للعزل، كطبقة دفاع ثانية حتى لو نسي مطوّر إضافته يدوياً.
- اختبارات العزل تتأكّد أن مستخدم Tenant A لا يصل لموارد Tenant B (يحصل على `404`).

> **مخطّط Prisma:** يتطلّب migration جديدة لإضافة `passwordHash` و`AccountProfile.tenantId` والحقول الزمنية الجديدة في هذه المرحلة فقط. أما حقول `Subscription` (`trialEndsAt`, `currentPeriodEnd`, `cancelAtPeriodEnd`) و enum `SubscriptionStatus` فهي موجودة من خطة الأساس (Sprint 0) ولا تتطلّب migration هنا — هذه المرحلة تنشئ فقط صفّ `Subscription` بحالة `trialing` عند التسجيل. (التنفيذ خارج نطاق هذه الوثيقة — تُسلَّم لـbackend.)

## 8. خارج النطاق (Out of Scope)

- **الفوترة الفعلية** (بوّابة الدفع، الترقية، انتهاء التجربة فعلياً) — **المرحلة 6**. هنا يُنشأ `Subscription` بحالة `trialing` فقط، دون أي تكامل دفع.
- **الـUI / الواجهة الأمامية** — **المرحلة 7**.
- **ربط API الفعلي مع LinkedIn / X** (OAuth، نشر فعلي) — مرحلة لاحقة. `AccountProfile` وصفي فقط.
- **الأدوار والصلاحيات المتعدّدة** (RBAC، دعوة أعضاء للفريق) — خارج هذه المرحلة.
- **التحقّق من البريد، إعادة تعيين كلمة المرور، MFA** — مرحلة لاحقة.
- **حقوق PDPL (تصدير/حذف البيانات — NFR-5):** **داخل نطاق هذه المرحلة** عبر `POST /me/export` و`DELETE /me` (انظر 4.2). يبقى خارج النطاق هنا فقط: تفاصيل تنفيذ المهمة المجدولة للـpurge وفترة الاحتجاز (تُسلَّم لـbackend/devops).

## 9. معايير القبول (Acceptance Criteria)

- [ ] `POST /auth/register` ببيانات صحيحة ينشئ `Tenant` + `User` + `Subscription(trialing)` مع `trialEndsAt = now + 7d` في معاملة ذرّية، ويعيد `AuthTokens` صالحة.
- [ ] التسجيل ببريد مستخدم مسبقاً يُرجِع `409 EMAIL_ALREADY_EXISTS` مع رسالة ترشد للدخول (`US-1.1`)، ولا ينشئ أي سجلّ.
- [ ] `POST /auth/login` ببيانات صحيحة يعيد `AuthTokens`؛ وببيانات خاطئة يعيد `401 INVALID_CREDENTIALS` دون كشف وجود البريد.
- [ ] `POST /auth/refresh` بـrefresh token صالح يعيد زوجاً جديداً (rotation)؛ وبـtoken منتهٍ/غير صالح يعيد `401`.
- [ ] `GET /me` بـaccess token صالح يعيد المستخدم + الـTenant + ملخّص الاشتراك؛ وبدون token يعيد `401`؛ وبـtoken منتهٍ يعيد `401 TOKEN_EXPIRED`.
- [ ] كلمة المرور تُخزَّن كـhash (`argon2`/`bcrypt`) ولا تظهر أبداً في أي استجابة، ولا `passwordHash`.
- [ ] CRUD لـ`/accounts` يعمل بالكامل، وكل عملية مُقيّدة بـ`tenantId` المستخرج من الـJWT.
- [ ] مستخدم Tenant A لا يستطيع قراءة/تعديل/حذف `AccountProfile` يخصّ Tenant B (يحصل على `404`، لا `403`).
- [ ] `tenantId` لا يُقرأ أبداً من جسم الطلب أو الـquery — فقط من الـJWT المُتحقَّق منه.
- [ ] `POST /me/export` بـaccess token صالح يعيد `200` مع حِزمة JSON تضمّ بيانات الـTenant الحالي (المستخدمون بدون `passwordHash`، `BrandProfile`، `Posts`، `AccountProfile`، الاشتراكات)، ولا تتضمّن أي سجلّ يخصّ tenant آخر — يحقّق NFR-5 (حقّ التصدير).
- [ ] `DELETE /me` مع `confirm: true` يُجري soft-delete فورياً (وسم `deletedAt` + إبطال الجلسات) ويعيد `202` مع `purgeAfter`، ويجدول الـpurge النهائي؛ وبدون `confirm: true` يعيد `422 CONFIRMATION_REQUIRED` دون أي حذف — يحقّق NFR-5 (حقّ الحذف).
- [ ] كل الأخطاء تتبع `ErrorEnvelope` الموحّد عبر `GlobalExceptionFilter`.
- [ ] كل المعرّفات وأسماء الحقول والمسارات بالإنجليزية.
