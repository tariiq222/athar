# دماغ الشركة / التهيئة — أثر (Spec)

> المرحلة التي تبني `BrandProfile`: المرجع الذي يقرأه محرّك المحتوى في كل توليد لاحق ([16-معمارية-المحرّك.md](../../16-معمارية-المحرّك.md)). الهدف: من رابط موقع وحسابات → ملف علامة عميق قابل للتعديل، بأقل إدخال يدوي. يغطّي **FR-1، FR-2، FR-3** و **US-2.1، US-2.2، US-2.3**.
>
> أسماء الأنواع والواجهات والمسارات بالإنجليزية (code-facing). الشرح بالعربي.

## نظرة عامة

التهيئة (onboarding) هي خطوة "تعليم النظام عن الشركة" مرّة واحدة، ونتيجتها `BrandProfile` واحد لكل عميل (tenant). تتكوّن من ثلاث حلقات:

1. **تحليل آلي (analyze):** العميل يدخل رابط موقعه ومعرّفات حساباته → النظام يجلب المحتوى ويلخّصه عبر `ContentProvider` إلى مسوّدة ملف (`BrandProfileDraft`) فيها نبرة/منتجات/جمهور/كلمات مفتاحية، **ويقترح محاور** مناسبة لمجال العميل.
2. **تأكيد (confirm):** النظام يطرح أسئلة قصيرة (نبرة، ممنوعات، منافسون، أهداف) **والعميل يحدّد محاوره** (اقتراحات النظام مجرّد نقطة بداية قابلة للإضافة/الحذف).
3. **تثبيت (commit):** يُبنى `BrandProfile` + `brandKit` ويُخزّن كمرجع دائم لكل توليد لاحق (FR-3).

التحليل **مساعد لا حاكم**: مدخلات العميل هي الأساس، والنظام يقترح فقط. عند فشل الجلب يُسمح بالإدخال اليدوي الكامل (US-2.1).

## المبادئ

- **العميل يقود، النظام يقترح:** التحليل الآلي يملأ مسوّدة قابلة للتعديل بالكامل؛ لا يُثبّت شيء بلا تأكيد العميل. المحاور خصوصاً يحدّدها العميل والنظام يقترح فقط.
- **التحليل خلف interface:** تلخيص الموقع/الحسابات يمرّ عبر `ContentProvider` فقط (نفس المزوّد النصّي في ملف ١٦)؛ لا استدعاء نموذج مباشر.
- **الجلب خلف interface:** سحب صفحة الموقع وملف الحساب يمرّ عبر `SearchProvider`/fetch؛ لا fetch منثور في الكود.
- **الفشل لا يوقف التهيئة:** تعذّر جلب الموقع → إعلام + إدخال يدوي؛ التهيئة تكتمل بمدخلات العميل وحدها (US-2.1).
- **تقليل البيانات (PDPL):** يُجلب من المصادر العامة فقط ما يلزم لبناء الملف؛ لا تخزين صفحات خام بلا داعٍ، وموافقة صريحة عند بدء التهيئة (NFR-5).
- **multi-tenant منطقي:** كل صفّ يحمل `tenantId`؛ كل قراءة/كتابة مقيّدة بـtenant العميل الحالي.
- **التكلفة كقيد:** عدد عمليات الجلب/التلخيص لكل تحليل محدود؛ كل استدعاء مزوّد يُسجّل في `UsageRecord`.

## أنواع البيانات (TypeScript)

> `BrandProfile`، `BrandKit`، `AccountProfile` معرّفة في Prisma (مصدر الحقيقة) وفي ملف ١٦. هنا نبني فوقها أنواع **هذه المرحلة فقط**.

```ts
type Platform = 'linkedin' | 'x'

// مدخل التهيئة الأوّلي (من العميل)
interface OnboardingInput {
  websiteUrl?: string          // قد يغيب → مسار يدوي
  accounts: AccountInput[]      // معرّفات/روابط الحسابات للتخصيص (بلا ربط API)
  consentAccepted: boolean      // موافقة PDPL — شرط لبدء التحليل
}
interface AccountInput {
  platform: Platform
  handle?: string              // @handle أو رابط الملف العام
}

// نتيجة التحليل الآلي (مسوّدة، غير مثبّتة)
interface BrandAnalysisResult {
  source: 'website' | 'accounts' | 'mixed' | 'manual'
  fetchStatus: FetchStatus     // نجاح/فشل الجلب لكل مصدر
  tone: string                 // النبرة المستخلصة (وصف)
  products: string[]           // منتجات/خدمات مكتشفة
  audience: string             // الجمهور المرجّح
  keywords: string[]           // كلمات مفتاحية مستخلصة
  suggestedTopics: string[]    // محاور يقترحها النظام لمجال العميل (FR-2)
  suggestedCompetitors: string[] // منافسون محتملون (اقتراح فقط)
  confidence: number           // 0..1 — جودة التحليل الكلية
  notes: string[]              // ملاحظات/تحذيرات (مثل: تعذّر جلب الموقع)
}

interface FetchStatus {
  website?: 'ok' | 'failed' | 'skipped'
  accounts: { platform: Platform; status: 'ok' | 'failed' | 'skipped' }[]
}

// مسوّدة الملف المعروضة للعميل قبل التثبيت (قابلة للتعديل بالكامل)
interface BrandProfileDraft {
  tone: string
  audience: string
  goals: string
  topics: string[]             // ما اعتمده العميل (أساسه اقتراحات النظام + إضافاته)
  prohibitions: string[]
  competitors: string[]
  keywords: string[]
  brandKit: BrandKitDraft
}

interface BrandKitDraft {
  colors: string[]             // ألوان مستخلصة من الموقع أو يدوية
  logoUrl?: string             // شعار مكتشف أو مرفوع
  visualStyle: string          // وصف الستايل البصري للصور لاحقاً (يُحقن في gpt-image)
  font: string                 // افتراضي IBM Plex Sans Arabic
}

// سؤال تأكيد قصير يُطرح على العميل
interface ConfirmationQuestion {
  id: string
  field: 'tone' | 'prohibitions' | 'competitors' | 'goals' | 'topics'
  prompt: string               // نص السؤال بالعربي
  kind: 'single' | 'multi' | 'text'
  suggestions?: string[]       // قيم مقترحة (من BrandAnalysisResult) — قابلة للقبول/الرفض
  required: boolean
}

interface ConfirmationAnswer {
  questionId: string
  field: ConfirmationQuestion['field']
  value: string | string[]
}
```

## المكوّنات والواجهات

### الخدمة الداخلية

```ts
interface OnboardingService {
  // FR-1: جلب + تحليل آلي → مسوّدة غير مثبّتة
  analyze(input: OnboardingInput, tenantId: string): Promise<BrandAnalysisResult>

  // FR-2: اشتقاق أسئلة التأكيد من نتيجة التحليل
  buildQuestions(analysis: BrandAnalysisResult): ConfirmationQuestion[]

  // FR-2/FR-3: دمج التحليل + إجابات العميل → ملف مثبّت
  commit(
    draft: BrandProfileDraft,
    tenantId: string,
    accounts: AccountInput[]
  ): Promise<BrandProfile>
}
```

- `analyze` يستهلك `ContentProvider` (تلخيص) + `SearchProvider`/fetch (جلب)؛ يُسجّل كل استدعاء في `UsageRecord`.
- `buildQuestions` نقي (pure): يحوّل الثقة المنخفضة/الحقول الناقصة إلى أسئلة؛ المحاور دائماً سؤال (`required`) لأن العميل يقودها.
- `commit` يكتب `BrandProfile` + `AccountProfile[]` بـ`tenantId`، ويهيّئ `learnedPreferences=''`.

### مسارات REST

| Method | Path | الغرض | المدخل | المخرج |
|---|---|---|---|---|
| `POST` | `/brand/analyze` | تحليل مبدئي من موقع/حسابات (FR-1) | `OnboardingInput` | `{ analysis: BrandAnalysisResult, questions: ConfirmationQuestion[] }` |
| `POST` | `/brand/profile` | إنشاء الملف من المسوّدة بعد التأكيد (FR-2/FR-3) | `BrandProfileDraft` + `accounts` | `BrandProfile` |
| `GET` | `/brand/profile/:id` | قراءة الملف (مرجع/سياق) | — | `BrandProfile` |
| `PATCH` | `/brand/profile/:id` | تعديل شامل (نبرة/ممنوعات/**محاور**…) (US-2.3) | `Partial<BrandProfileDraft>` | `BrandProfile` |

- كل المسارات تُقيّد بـ`tenantId` المستخرج من جلسة العميل؛ `:id` يجب أن يخصّ نفس الـtenant وإلا `404`.
- `POST /brand/analyze` يُرجع `analysis` و`questions` معاً ليبدأ الـUI خطوة التأكيد فوراً.
- `POST /brand/profile` لا يستقبل نتيجة التحليل خام بل **المسوّدة المعتمدة** من العميل (مبدأ "العميل يقود").
- `PATCH` يقبل أي مجموعة جزئية؛ تعديل `topics` هو المسار الرسمي لإعادة ضبط محاور البحث لاحقاً.

## التدفّقات

### التدفّق الأساسي (مع موقع صالح)

1. العميل يقبل الموافقة (PDPL) ويدخل `websiteUrl` + `accounts` → `POST /brand/analyze`.
2. `analyze` يجلب الموقع/ملفات الحسابات عبر `SearchProvider`/fetch (ضمن سقف عدد الطلبات).
3. المحتوى المجلوب يُمرَّر إلى `ContentProvider` للتلخيص → `BrandAnalysisResult` (نبرة، منتجات، جمهور، كلمات، **محاور مقترحة**، ألوان/شعار للـ`brandKit`).
4. `buildQuestions` يشتقّ `ConfirmationQuestion[]` (نبرة، ممنوعات، منافسون، أهداف، **محاور**).
5. الـUI يعرض الأسئلة؛ العميل يجيب ويعدّل، **ويحدّد محاوره** (يقبل/يرفض/يضيف على اقتراحات النظام).
6. الإجابات تُدمج مع التحليل في `BrandProfileDraft` → `POST /brand/profile` → `commit` يثبّت `BrandProfile` + `AccountProfile[]`.
7. الملف صار مرجعاً دائماً يقرأه المحرّك في كل توليد (FR-3).

### تدفّق فشل الجلب (US-2.1)

1. `analyze` يفشل في جلب الموقع → `fetchStatus.website='failed'` و`notes` يوضّح السبب.
2. ما أمكن جلبه من الحسابات يُلخّص؛ وإلا `source='manual'` و`confidence` منخفضة.
3. `buildQuestions` يحوّل الحقول الفارغة إلى أسئلة إدخال يدوي (`kind:'text'`, `required:true`).
4. العميل يملأ الملف يدوياً ويكمل من الخطوة 6 أعلاه — **التهيئة تنجح بلا أي جلب آلي**.

### تدفّق التعديل اللاحق (US-2.3)

- `GET /brand/profile/:id` لعرض الحالي → تعديل → `PATCH /brand/profile/:id`؛ يشمل إعادة ضبط `topics`/`prohibitions`/`tone`. التغيير ساري على التوليد التالي مباشرة.

## معالجة الأخطاء

| الموضع | الفشل | السلوك |
|---|---|---|
| جلب الموقع | رابط غير صالح / تعذّر الوصول | `fetchStatus.website='failed'` + `notes`؛ السماح بالإدخال اليدوي (لا توقف) |
| جلب الحساب | ملف خاص/غير موجود | `status='failed'` لذلك الحساب؛ المتابعة بالبقية |
| التلخيص | خطأ `ContentProvider` | إعادة محاولة محدودة → عند الفشل: مسوّدة بحد أدنى + `notes`؛ تسجيل في `UsageRecord` |
| الموافقة | `consentAccepted=false` | رفض `POST /brand/analyze` بـ`422` (شرط PDPL قبل أي جلب) |
| التثبيت | حقول إلزامية ناقصة (نبرة/محاور) | `422` مع تحديد الحقول؛ لا إنشاء ملف ناقص |
| العزل | `:id` لا يخصّ الـtenant | `404` (لا تسريب وجود صفّ عبر tenants) |
| السقف | تجاوز عدد عمليات الجلب/التلخيص | إيقاف التحليل عند الحد + `notes`؛ المسوّدة الجزئية تُعرض |

## الاعتمادات

- `ContentProvider` (من ملف ١٦) — تلخيص النبرة/المنتجات/الجمهور/الكلمات واقتراح المحاور.
- `SearchProvider`/fetch — جلب صفحة الموقع وملفات الحسابات العامة (بلا ربط API للمنصّات).
- نموذج Prisma: `Tenant`, `User`, `BrandProfile`, `AccountProfile` (مصدر الحقيقة، لا يُكرّر هنا).
- `UsageRecord` — تسجيل استدعاءات المزوّدين للتكلفة.
- إطار الموافقة/PDPL — تخزين الموافقة وقتها لتقليل البيانات وحق الحذف لاحقاً (NFR-5).
- ملف ١٦ (محرّك المحتوى) — المستهلك النهائي لـ`BrandProfile`.

## معايير القبول

- **AC-1 (FR-1):** بإدخال رابط موقع صالح، يُرجع `POST /brand/analyze` نبرة/منتجات/جمهور/كلمات مفتاحية مبدئية في `BrandAnalysisResult`.
- **AC-2 (FR-1/US-2.1):** عند تعذّر جلب الموقع، يُرجع `fetchStatus.website='failed'` ولا يفشل المسار؛ التهيئة قابلة للإكمال يدوياً.
- **AC-3 (FR-2):** يقترح النظام `suggestedTopics` مناسبة لمجال العميل، **والعميل يستطيع إضافة/حذف/تعديل** محاوره، والمعتمد فقط يُخزّن في `BrandProfile.topics`.
- **AC-4 (FR-2):** تُطرح أسئلة تأكيد قصيرة تغطّي النبرة والممنوعات والمنافسين والأهداف، وإجاباتها تُدمج في الملف.
- **AC-5 (FR-3):** بعد `POST /brand/profile`، يُخزّن `BrandProfile` (شاملاً المحاور و`brandKit`) ويُقرأ عبر `GET /brand/profile/:id`.
- **AC-6 (US-2.3):** `PATCH /brand/profile/:id` يعدّل أي حقل (شاملاً المحاور) ويسري على التوليد التالي.
- **AC-7 (multi-tenant):** كل صفوف الملف/الحسابات تحمل `tenantId`، وأي وصول لـ`:id` خارج الـtenant يُرجع `404`.
- **AC-8 (PDPL):** بدون `consentAccepted=true` يُرفض التحليل (`422`)؛ لا يُجلب أو يُخزّن أكثر مما يلزم لبناء الملف.
