# المرحلة ٧ — الواجهة الأمامية (Frontend Spec) — أثر

> واجهة العميل الكاملة لمنصّة "أثر" (محتوى عربي). تستهلك خدمات الـbackend عبر عقود REST (NestJS) وتعرضها بتجربة **عربية-أوّل، RTL كامل، موبايل-أوّل**. لا منطق أعمال في الواجهة — التوليد والبحث والفوترة والاستشهاد كلّها خلف الـAPI. تغطّي هذه المرحلة شاشات المراحل ١–٦ بصرياً وتربطها بعقودها. تطبّق حدود المنصات من [15-مواصفات-المنصات.md](../../15-مواصفات-المنصات.md) مرئياً عبر `CharCounter`، وتعرض مخرجات محرّك [16-معمارية-المحرّك.md](../../16-معمارية-المحرّك.md).
>
> أسماء المكوّنات والمسارات والأنواع والـhooks بالإنجليزية (code-facing). كل النصوص الظاهرة للمستخدم عربية. الشرح هنا بالعربي.
>
> **ملاحظة على العقود:** عقود الـAPI تنضج مع بناء المراحل ١–٦. التواقيع الواردة هنا توجيهية تمثّل الشكل المتوقّع؛ **العقد الدقيق لكل نطاق يُثبّت مع تسليم مرحلته** ويُحدّث هذا الملف عند التثبيت.

## المبادئ
- **عربي-أوّل، RTL كامل:** `dir="rtl"` و`lang="ar"` على الجذر؛ كل تخطيط يُصمّم RTL أصالةً لا كقلب لاحق. خط `IBM Plex Sans Arabic` افتراضي للنظام كلّه.
- **موبايل-أوّل:** التصميم يبدأ من عرض الموبايل ثم يتوسّع؛ أهداف اللمس ≥٤٤px، تنقّل سفلي على الموبايل، تخطيطات عمود واحد تترقّى لشبكات على الشاشات الأكبر.
- **لا منطق أعمال في الواجهة:** الواجهة تعرض وتجمع المدخلات وتستدعي العقود فقط. التوليد، النقد الذاتي، الاستشهاد، حدود المنصة كقواعد إلزامية، حساب الفوترة — كلّها في الـbackend. الواجهة تعكس الحدود **بصرياً** (CharCounter) كإرشاد لا كمصدر حقيقة.
- **العقد هو الحدّ الفاصل:** كل تفاعل بيانات يمرّ عبر طبقة `services/` ثم `hooks/`؛ لا استدعاء `fetch` مباشر من مكوّن.
- **حالة الخادم منفصلة عن حالة الواجهة:** بيانات الخادم تُدار عبر طبقة جلب/تخزين مؤقّت (server state)؛ حالة الواجهة المحلّية (مسوّدات، نماذج، فتح/إغلاق) منفصلة.
- **حالات صريحة دائماً:** كل شاشة تعرّف صراحةً حالات التحميل والفراغ والخطأ، لا فقط الحالة السعيدة (انظر قسم الأخطاء والحالات الفارغة).
- **مكوّنات مشتركة موحّدة:** حالة البوست تُعرض حصراً عبر `StatusBadge`؛ لا تكرار لمنطق العرض في الصفحات.

## التقنية والبنية
- **Next.js (App Router) + React + TypeScript.** توجيه عبر مجلّدات `app/`.
- **التنسيق:** نظام تصميم موحّد (tokens للألوان والمسافات والطباعة) — لا ألوان Tailwind خام ولا `dark:` منثورة في الصفحات؛ الألوان عبر tokens.
- **الطباعة:** `IBM Plex Sans Arabic` محمّل عبر `next/font` بأوزان عربية، مع fallback نظامي عربي.
- **الحالة:** طبقة server-state للجلب/التخزين المؤقّت/إعادة الجلب؛ سياقات React خفيفة للجلسة واللغة والإشعارات.

### تفاصيل الشاشات

#### ١–٢ تسجيل/دخول — `/login`، `/register` (المرحلة ٣)
- نموذج بريد/كلمة مرور؛ تحقّق حقول فوري؛ زر إرسال بحالة تحميل.
- نجاح → حفظ JWT (انظر طبقة الـAPI) → تحويل إلى `/onboarding` إن لم يكتمل البروفايل، وإلا `/dashboard`.
- روابط متبادلة بين الدخول والتسجيل؛ رسائل خطأ عربية واضحة (بيانات خاطئة، حساب موجود).

#### ٣ Onboarding / دماغ الشركة — `/onboarding` (المرحلة ٢)
معالج (wizard) متعدّد الخطوات يبني `BrandProfile`:
1. **إدخال:** حقل موقع الشركة + حسابات التواصل (LinkedIn / X).
2. **عرض التحليل:** عرض ما استنتجه الـbackend (محاور مقترحة، نبرة، جمهور) في حالة قراءة، مع مؤشّر تقدّم أثناء التحليل (قد يكون async).
3. **أسئلة تأكيد:** أسئلة يجيب عليها العميل لتدقيق الاستنتاج.
4. **تحديد/اعتماد المحاور:** اختيار/إضافة/حذف `topics`، تعريف الممنوعات `prohibitions`.
5. **عرض `BrandProfile` النهائي** واعتماده → إنشاء/تحديث البروفايل عبر العقد → تحويل إلى `/dashboard`.
- المعالج يحفظ تقدّمه؛ الخروج والعودة يستأنف من آخر خطوة.

#### ٤ اللوحة — `/dashboard` (المرحلة ١/٤)
- نظرة سريعة: عدد البوستات بكل حالة، أقرب البوستات المجدولة، اختصار "ولّد بوست" و"خطة شهر".
- يستهلك ملخّصاً من `posts`/`calendar`.

#### ٥ قائمة البوستات — `/posts` (المرحلة ١/٤)
- قائمة `PostCard` مع تصفية حسب الحالة (`StatusBadge`) والمنصة.
- نقر بطاقة → `/posts/[postId]`.

#### ٦ محرّر البوست — `/posts/[postId]` (المراحل ١/٤)
- يعرض مخرجات المحرّك (`Post`): النص + `SourceList` (الاستشهادات) + `ImagePreview`.
- تعديل النص داخل محرّر مع `CharCounter` يطبّق حدّ المنصة بصرياً.
- عرض `issues` من النقد الذاتي إن مُرّرت أفضل نسخة دون اجتياز (شفافية الجودة).
- إجراءات: حفظ تعديل، اعتماد (`approve` → الحالة تنتقل إلى `approved`)، إعادة توليد (يستدعي عقد `content`).
- عند الاعتماد بعد تعديل: يُرسل الفرق للـbackend لالتقاط `learnedPreferences` (منطق التلخيص في الـbackend).

#### ٧ توليد — `/generate` (المرحلة ١)
- **بوست مفرد:** اختيار منصة + نوع محتوى + محور/brief اختياري → زر "ولّد بوست" → ينتظر `Post` → تحويل إلى محرّره.
- **خطة شهر:** زر "خطة شهر" يطلق job غير متزامن (BullMQ في الـbackend) → **مؤشّر تقدّم async** يتابع الحالة عبر polling/stream → عند الاكتمال يعرض ملخّص الخطة وروابط للتقويم.
- البوستات الفاشلة ضمن الخطة تُعلّم ولا تُسقط الخطة (تعرضها الواجهة بحالة خطأ قابلة لإعادة المحاولة).

#### ٨ التقويم — `/calendar` (المرحلة ٤)
- `CalendarGrid` شهري يعرض المناسبات السعودية (مواسم/أعياد) كطبقة خلفية، والبوستات المجدولة كعناصر داخل الأيام.
- **سحب/إفلات اختياري** لإعادة الجدولة (يُحدّث `scheduledAt` عبر العقد)؛ على الموبايل بديل: فتح اليوم وتعديل التاريخ عبر منتقي.
- نقر بوست → محرّره أو شاشة نشره.

#### ٩ النشر اليدوي — `/publish/[postId]` (المرحلة ٥)
- لكل منصة: **نسخ النص بضغطة** (clipboard) + **تنزيل الصورة** + زر **فتح المنصة** (deep link لتطبيق/موقع المنصة) + زر **تعليم "تم النشر"** (يحدّث الحالة إلى `published`).
- لا نشر تلقائي (خارج النطاق) — المستخدم ينشر يدوياً ثم يعلّم.
- تأكيد بصري بعد كل إجراء (نسخ ناجح، تنزيل ناجح).

#### ١٠ الاشتراك والفوترة — `/billing` (المرحلة ٦)
- عرض الباقة الحالية والحدّ المتبقّي (بوستات/شهر) من عقد `billing`.
- ترقية/اشتراك عبر **Moyasar** (الواجهة تعرض نموذج/إعادة توجيه الدفع؛ منطق الدفع والتحقّق في الـbackend).
- عرض الفاتورة/سجلّ الفواتير.
- تجاوز سقف الباقة → حالة محظورة واضحة مع دعوة للترقية (انظر الأخطاء).

#### ١١–١٢ الإعدادات وملفات الحسابات — `/settings`، `/settings/accounts`
- إعدادات الحساب: بيانات المستخدم، اللغة (ar/en)، تسجيل خروج.
- ملفات الحسابات: ربط/إدارة حسابات LinkedIn / X المستخدمة في الـonboarding والنشر.

- `CharCounter` يستخدم نفس منطق العدّ المرجعي (مكتبة `twitter-text` لـX) ليطابق ما يفرضه الـbackend، فلا يفاجأ المستخدم برفض الخادم.

  - منطق الاشتقاق يعيش في الـfrontend (دالة تحويل `PostSummary → StatusBadgeKind`)، فلا تُمرّر `'scheduled'`/`'failed'` كأنها قيمة `PostStatus` قادمة من الـAPI.

## استهلاك الـAPI (طبقة `services/` + `hooks/`)

كل نطاق له ملف service يلفّ عقود مرحلته، وhook يغلّفه بحالة server-state (تحميل/خطأ/تخزين مؤقّت/إعادة جلب). لا `fetch` مباشر من المكوّنات.

| النطاق | service | hooks (أمثلة) | يستهلك مرحلة |
|---|---|---|---|
| المصادقة | `services/auth.ts` | `useAuth`, `useSession` | ٣ |
| دماغ الشركة | `services/brand.ts` | `useBrandProfile`, `useBrandAnalysis` | ٢ |
| المحتوى/التوليد | `services/content.ts` | `useGeneration`, `useMonthPlan` (async progress) | ١ |
| البوستات | `services/posts.ts` | `usePosts`, `usePost`, `useApprovePost` | ١/٤ |
| التقويم | `services/calendar.ts` | `useCalendar`, `useReschedule` | ٤ |
| التصدير/النشر اليدوي | `services/export.ts` | `useExportAssets` (نص/صورة) | ٥ |
| الفوترة | `services/billing.ts` | `useBilling`, `useSubscription`, `useInvoices` | ٦ |

### المصادقة (JWT)
- بعد الدخول يُخزّن JWT ويُحقن في رأس `Authorization: Bearer` عبر `apiClient`.
- `AppShell` يتحقّق من وجود/صلاحية الجلسة قبل عرض مجموعة `(app)`؛ غياب/انتهاء → `/login`.
- 401 من أي عقد → محاولة تحديث (إن وُجد refresh)، وإلا إخراج وتحويل للدخول مع حفظ المسار المقصود للعودة.
- آلية التخزين الدقيقة (cookie httpOnly مقابل تخزين العميل) **تُثبّت مع عقد المرحلة ٣**؛ تُفضّل httpOnly cookie أمنياً.

## الحالة والتوجيه
- **server state:** عبر طبقة جلب/تخزين مؤقّت لكل hook (مفاتيح cache لكل نطاق: `posts`, `post:[id]`, `calendar:[month]`, `billing` ...). إبطال موجّه بعد الطفرات (approve/reschedule/generate).
- **client state:** سياقات خفيفة فقط: `SessionContext` (المستخدم/JWT)، `LocaleContext` (ar/en، الاتجاه)، `ToastContext` (إشعارات). مسوّدات المحرّر والنماذج حالة محلّية في الصفحة.
- **التوجيه:** App Router؛ مجموعة `(auth)` عامة، `(app)` محميّة. التحويلات: دخول→onboarding/dashboard، توليد مفرد→محرّر البوست، اكتمال خطة الشهر→التقويم.
- **التوليد غير المتزامن:** خطة الشهر تتابَع عبر `getMonthPlanProgress` (polling أو stream)؛ مؤشّر التقدّم في `/generate` يبقى حيّاً عبر التنقّل (الحالة في server-state بمفتاح `jobId`).

## RTL و i18n
- **الاتجاه:** `<html dir="rtl" lang="ar">`؛ كل التخطيطات منطقية (`start`/`end` لا `left`/`right`). الافتراضي ar؛ en متاح كثانوي يقلب الاتجاه إلى ltr.
- **الخط:** `IBM Plex Sans Arabic` عبر `next/font` (أوزان عربية)، مع fallback نظامي عربي. الأرقام: أرقام عربية شرقية أو غربية بحسب تفضيل موحّد عبر `formatNumber` (مصدر واحد، لا تنسيق منثور).
- **مرايا الأيقونات:** الأيقونات الاتجاهية (رجوع/تقدّم، أسهم، إرسال) تُعكس في RTL؛ الأيقونات غير الاتجاهية (بحث، إعدادات) لا تُعكس.
- **ترتيب التركيز (focus order):** يتبع التدفّق المنطقي RTL (يمين→يسار، أعلى→أسفل)؛ تسلسل tab صحيح في النماذج والمعالج والتقويم؛ حلقات تركيز ظاهرة.
- **النصوص:** كل النصوص الظاهرة من ملفّات i18n (`i18n/ar`, `i18n/en`)؛ لا نص عربي مكتوب داخل الكود مباشرة (يسهّل المراجعة والـen لاحقاً). المعرّفات/المفاتيح إنجليزية.
- **التواريخ والمناسبات:** التقويم يعرض الأسبوع السعودي والمناسبات السعودية؛ تنسيق التاريخ موحّد عبر helper.

## الأخطاء والحالات الفارغة وحالات التحميل

| الموضع | الحالة | سلوك الواجهة |
|---|---|---|
| Onboarding | غير مكتمل | اللوحة/التوليد محجوبة بدعوة "أكمل دماغ الشركة" → `/onboarding` يستأنف من آخر خطوة. |
| قائمة البوستات | لا بوستات | حالة فارغة عربية + زر "ولّد أول بوست". |
| التوليد المفرد | فشل توليد | رسالة خطأ عربية + زر "أعد المحاولة"؛ لا يفقد المدخلات. |
| خطة الشهر | بوست فاشل ضمن الخطة | يُعلّم بحالة `failed` (StatusBadge) قابل لإعادة التوليد؛ بقية الخطة سليمة. |
| محرّر البوست | نقد لم يجتَز بعد ٣ دورات | يعرض أفضل نسخة + `issues` كتنبيه شفّاف (لا يحجب الاعتماد). |
| الصورة | فشل توليد/تكسّر | `ImagePreview` يعرض البديل (overlay) أو "نص فقط" بعلامة واضحة. |
| الفوترة | تجاوز سقف الباقة | حظر التوليد بحالة واضحة "بلغت حدّ باقتك" + دعوة ترقية → `/billing`. |
| الجلسة | انتهاء التجربة/الاشتراك | لافتة عبر التطبيق + تحويل مسارات التوليد إلى `/billing`؛ القراءة تبقى متاحة. |
| المصادقة | 401/انتهاء JWT | إخراج لطيف → `/login` مع حفظ المسار للعودة. |
| أي جلب | تحميل | حالة تحميل صريحة (هيكل/skeleton) لكل قائمة وبطاقة وصورة. |
| أي جلب | خطأ شبكة | رسالة عربية + "أعد المحاولة"؛ لا شاشة بيضاء. |

> كل شاشة تعرّف **الحالات الأربع** (تحميل، فراغ، خطأ، نجاح) صراحةً — لا الحالة السعيدة وحدها.

## الموبايل-أوّل (تخطيطات أساسية)
- **الهيكل:** `AppShell` بتنقّل سفلي (`BottomNav`) على الموبايل، يترقّى لتنقّل جانبي (`SideNav`) على الشاشات الأكبر.
- **عمود واحد افتراضاً:** القوائم والبطاقات والمعالج عمود واحد على الموبايل، تترقّى لشبكات/أعمدة على ≥md.
- **التقويم:** شبكة شهرية على الديسكتوب؛ على الموبايل **قائمة أيام عمودية** (لا شبكة مكتظّة)، وإعادة الجدولة عبر منتقي تاريخ بدل السحب.
- **المحرّر:** النص بعرض كامل، الأدوات (CharCounter/إجراءات) في شريط لاصق سفلي على الموبايل.
- **أهداف اللمس ≥٤٤px** وتباعد كافٍ؛ لا تفاعل يعتمد على hover فقط.

## معايير القبول
- [ ] كل الشاشات الـ١٢ موجودة بمساراتها (App Router) وتُحمّل بلا خطأ، ومحميّة بحارس JWT في مجموعة `(app)`.
- [ ] `dir="rtl"`/`lang="ar"` على الجذر؛ كل التخطيطات منطقية (start/end)؛ خط `IBM Plex Sans Arabic` مطبّق على النظام كلّه.
- [ ] كل النصوص الظاهرة عربية ومن ملفّات i18n؛ لا نص عربي داخل الكود؛ كل المعرّفات/المسارات/الأنواع إنجليزية.
- [ ] حالة البوست تُعرض حصراً عبر `StatusBadge` (لا تكرار منطق الحالة)، وله accessible name عربي لا يعتمد على اللون وحده.
- [ ] `CharCounter` يطابق حدود ملف ١٥ بصرياً (X بعدّ `twitter-text` لا `.length`، LinkedIn ٣٬٠٠٠) ولا يدّعي فرض الحدّ (الفرض في الـbackend).
- [ ] لا `fetch` مباشر من مكوّن؛ كل وصول بيانات عبر `services/` + `hooks/`، وكل النطاقات السبعة ممثّلة.
- [ ] JWT يُحقن في كل طلب؛ 401 يُعالج بإخراج لطيف مع حفظ المسار للعودة.
- [ ] كل شاشة تعرّف الحالات الأربع (تحميل/فراغ/خطأ/نجاح)؛ لا شاشة بيضاء عند الخطأ.
- [ ] حالات الحوافّ مغطّاة: onboarding غير مكتمل، لا بوستات، فشل توليد، فشل بوست ضمن خطة الشهر، تجاوز سقف الباقة، انتهاء التجربة.
- [ ] خطة الشهر تعرض **مؤشّر تقدّم async** يبقى عبر التنقّل، والبوست الفاشل ضمنها قابل لإعادة التوليد دون إسقاط الخطة.
- [ ] النشر اليدوي يوفّر: نسخ بضغطة + تنزيل صورة + فتح المنصة + تعليم "تم النشر"؛ لا نشر تلقائي.
- [ ] التقويم يعرض المناسبات السعودية + البوستات المجدولة، مع بديل موبايل (قائمة أيام + منتقي) للسحب/الإفلات.
- [ ] موبايل-أوّل: تنقّل سفلي، عمود واحد، أهداف لمس ≥٤٤px، لا اعتماد على hover.
- [ ] أيقونات اتجاهية معكوسة في RTL؛ ترتيب تركيز منطقي RTL مع حلقات تركيز ظاهرة.
- [ ] لا ألوان Tailwind خام ولا `dark:` في الصفحات؛ الألوان عبر design tokens.

## ما هو خارج هذا الـspec (خارج النطاق)
- **منطق الأعمال كلّه في الـbackend:** التوليد، البحث والاستشهاد، النقد الذاتي، توليد الصور والتحقّق، حدود المنصة كقواعد إلزامية، حساب الباقات والفوترة، التلخيص لـ`learnedPreferences`. الواجهة تعرض وتجمع وتستدعي فقط.
- **النشر التلقائي للمنصات:** غير مدعوم؛ النشر يدوي (نسخ/تنزيل/فتح/تعليم) فقط.
- **الدفع نفسه (Moyasar):** منطق الدفع والتحقّق وويبهوك في الـbackend؛ الواجهة تعرض النموذج/إعادة التوجيه فقط.
- **العقود الدقيقة لكل نطاق:** تُثبّت مع تسليم مراحلها ١–٦؛ التواقيع هنا توجيهية.
