# المرحلة ٦ — الفوترة (Moyasar) — أثر (Spec)

> بوّابة الإيراد. اشتراك شهري/سنوي عبر **Moyasar** (mada / Apple Pay / بطاقات) بباقة واحدة عند الإطلاق، مع تجربة مجانية ٧ أيام، وتطبيق سقوف الباقة (مسودات/صور شهرياً) عبر `UsageRecord`، والتزام **ZATCA** للفوترة الإلكترونية. multi-tenant منطقي. يبني على نموذج Prisma المعرّف في الأساس (`Subscription`, `UsageRecord`, `Tenant`) ويضيف جدول `Invoice` فقط.
>
> أسماء الأنواع والمسارات بالإنجليزية (code-facing). الشرح بالعربي. يغطّي [FR-15](../../10-وثيقة-المتطلبات-PRD.md)، القصص US-1.2 / US-1.3، و NFR-5 / NFR-7.
>
> القاعدة الذهبية للدفع: **Moyasar هو مصدر الحقيقة، المتصفّح خصم افتراضي.** لا يُعتمد اشتراك إلا بعد إعادة جلب الدفعة من الخادم بالمفتاح السرّي. الـ`callback` للـUX، الـ`webhook` لدفتر الحساب.

## نظرة عامة

تأخذ هذه المرحلة العميل من نهاية التجربة المجانية إلى اشتراك مدفوع فعّال، وتُبقي وصوله محكوماً بسقوف الباقة:

- العميل يبدأ `trialing` عند التسجيل (المرحلة ٣ — Auth)، فيُولّد ويُصدّر بسقوف التجربة.
- عند طلب الاشتراك أو انتهاء التجربة → ينشئ النظام دفعة Moyasar، يوجّه العميل لإتمام mada/Apple Pay/بطاقة (مع تحدّي 3DS عند اللزوم).
- **webhook** من Moyasar (`payment_paid`) هو ما يحوّل `Subscription.status` فعلياً إلى `active` ويُصدر فاتورة ZATCA.
- بعدها كل توليد نصّي/صورة يستهلكه المحرّك (المرحلة ١) يُفحص ضد سقف الباقة عبر `canConsume(...)` قبل التنفيذ، ويُسجّل في `UsageRecord`.
- العميل يعرض اشتراكه، يُلغيه، ويحمّل فاتورته (US-1.3).

```
trialing ──(أول دفعة ناجحة)──▶ active ──(فشل تجديد)──▶ past_due ──(فشل نهائي/إلغاء)──▶ canceled
   │                                                          │
   └──(انتهاء التجربة بلا دفع)──▶ past_due (وصول معلّق)        └──(دفعة لاحقة ناجحة)──▶ active
```

## المبادئ

- **مصدر الحقيقة هو Moyasar، لا المتصفّح.** كل اعتماد اشتراك يمرّ بـ`GET /v1/payments/:id` بالمفتاح السرّي ومقارنة `status`/`amount`/`currency`/`metadata.tenant_id` قبل أي تفعيل. لا يُوثَق بـquery string الـ`callback` أبداً (NFR-4).
- **Fulfill on webhook, confirm on redirect.** الـ`callback` قد يضيع (تبويب مُغلق، انقطاع شبكة)؛ التفعيل الفعلي مربوط بـ`webhook` متكرّر الأمان (idempotent على `payment.id`). الـ`callback` يحسّن تجربة العميل فقط.
- **مفتاحان، جهتان.** `pk_*` في المتصفّح (إنشاء دفعة/توكن فقط)، `sk_*` لا يغادر الخادم أبداً. بيانات البطاقة لا تلمس خوادمنا (Moyasar.js / Apple Pay)؛ نبقى خارج نطاق PCI-DSS.
- **المبلغ بالوحدات الصغرى دائماً.** عدد صحيح بالهللة (`59900` = ٥٩٩٫٠٠ ريال)، لا أرقام عشرية، لا ضرب ×١٠٠ مكرّر.
- **Idempotency طرفان.** إرسال: `given_id` (UUIDv4) على كل `POST /payments`. استقبال: `payment.id` مفتاح فريد على الـ`webhook` فلا تفعيل مزدوج.
- **السقوف قيد على المحرّك، لا على الفوترة.** `canConsume(tenantId, kind)` تُقرأ قبل كل توليد لكل أنواع `UsageKind` الثلاثة (`text`/`image`/`search`)؛ التجاوز رفض واضح بلا استدعاء AI (حماية الهامش — NFR-7).
- **التجاوز وسط خطة الشهر لا يُسقط الخطة.** لو بلغ الـ`tenant` السقف أثناء توليد خطة شهر (٣٠ بوست)، البوستات المتبقّية تُعلَّم `skipped_quota` (حالة متميّزة عن فشل المزوّد)، والبوستات المكتملة تبقى، ويُبلَّغ العميل بالمتبقّي وسبب التوقّف — لا إسقاط للخطة ولا تراجع عمّا أُنجز.
- **الامتثال غير قابل للتفاوض.** كل فاتورة متوافقة مع **ZATCA** (رقم ضريبي، QR، حقول إلزامية) و**PDPL** (تقليل بيانات، حق الحذف) — NFR-5.
- **multi-tenant منطقي.** كل سجلّ (اشتراك، استهلاك، فاتورة) مقيّد بـ`tenantId`؛ لا تسرّب بين العملاء.

## الأنواع (Types)

```ts
// تعريف الباقة — ثابت في config، لا منثور في الكود
interface PlanDefinition {
  code: string              // 'business' (باقة الإطلاق الوحيدة)
  nameAr: string            // "أعمال"
  priceSar: number          // 599 — ريال (للعرض)
  priceMinor: number        // 59900 — هللة (للـ API؛ المصدر للدفع)
  annualPriceMinor: number  // سعر سنوي مخفّض (خصم)
  billingCycle: 'monthly' | 'annual'
  trialDays: number         // 7
  monthlyDraftCap: number   // سقف المسودّات شهرياً
  monthlyImageCap: number   // سقف الصور شهرياً
  monthlySearchCap: number  // سقف عمليات البحث شهرياً (NFR-7 — البحث استدعاء مكلف)
}

// سقوف التجربة المجانية — PlanDefinition منفصل بـ code='trial'.
// تُطبَّق طوال أيام التجربة الـ٧ (plan:'trial')، وهي أصغر من سقوف الباقة المدفوعة.
// المحرّك يقرأ نفس canConsume(...) سواء كان tenant في trialing أو active؛
// مصدر السقف هو PlanDefinition المطابق لباقة الاشتراك الحالي.
const TRIAL_PLAN: PlanDefinition = {
  code: 'trial',
  nameAr: 'تجربة مجانية',
  priceSar: 0,
  priceMinor: 0,
  annualPriceMinor: 0,
  billingCycle: 'monthly',
  trialDays: 7,
  monthlyDraftCap: 10,   // سقف تجربة أصغر — مثال؛ يُضبط في config
  monthlyImageCap: 5,    // سقف تجربة أصغر — مثال
  monthlySearchCap: 10,  // سقف تجربة أصغر — مثال
}

type SubscriptionStatus =
  | 'trialing'   // تجربة ٧ أيام فعّالة
  | 'active'     // مدفوع وفعّال
  | 'past_due'   // فشل دفع/تجديد أو انتهت التجربة بلا دفع → وصول معلّق
  | 'canceled'   // مُلغى (بطلب العميل أو فشل نهائي)

type UsageKind = 'text' | 'image' | 'search'   // مطابق UsageRecord.kind الموجود

// يطابق payload الـ webhook من Moyasar (مبسّط)
interface MoyasarWebhookEvent {
  id: string                // 'evt_…'
  type: MoyasarEventType
  created_at: string
  secret_token: string      // السرّ المشترك — يُقارن constant-time
  data: MoyasarPayment      // كائن الدفعة الكامل (لا يُوثق بلا re-fetch)
}

type MoyasarEventType =
  | 'payment_paid'
  | 'payment_failed'
  | 'payment_refunded'
  | 'invoice_paid'
  | 'invoice_expired'

interface MoyasarPayment {
  id: string                // 'payment_…' — مفتاح الـ idempotency
  status: 'initiated' | 'paid' | 'failed' | 'refunded'
  amount: number            // هللة — يُقارن === priceMinor المتوقّع
  currency: 'SAR'
  source: {
    type: 'creditcard' | 'applepay' | 'stcpay'
    company?: 'mada' | 'visa' | 'mastercard' | 'amex'
    transaction_url?: string   // وجهة تحدّي 3DS عند status='initiated'
    message?: string
  }
  metadata: { tenant_id: string; plan_code: string; cycle: string }
}

// فاتورة ZATCA متوافقة (تُخزَّن لكل دفعة ناجحة)
interface Invoice {
  id: string
  tenantId: string
  subscriptionId: string
  moyasarPaymentId: string  // الربط بالدفعة
  number: string            // تسلسلي لكل tenant
  issuedAt: string
  // مبالغ
  subtotalMinor: number     // قبل الضريبة
  vatRate: number           // 0.15
  vatAmountMinor: number
  totalMinor: number
  currency: 'SAR'
  // حقول ZATCA الإلزامية
  sellerName: string
  sellerVatNumber: string   // الرقم الضريبي للبائع (أثر)
  buyerName: string
  zatcaQr: string           // Base64 TLV (QR متوافق ZATCA) — انظر قسم ZATCA
  status: 'issued' | 'refunded'
}
```

## المكوّنات وواجهات REST

كل المسارات تحت `/billing` ومقيّدة بـ`tenantId` من سياق الـAuth (المرحلة ٣)، عدا الـ`webhook` (عام، يُحقّق بالسرّ المشترك).

### `POST /billing/subscribe`
ينشئ نيّة اشتراك ويبدأ تدفّق دفع Moyasar.
- **الإدخال:** `{ planCode: 'business', cycle: 'monthly' | 'annual' }`.
- **المنطق:** يقرأ `PlanDefinition`، يحسب `priceMinor` حسب الدورة، يُنشئ `given_id = uuidv5("sub:{tenantId}:{attempt}")` للـidempotency، ويعيد بيانات تهيئة Moyasar.js للواجهة (المرحلة ٧) مع `callback_url = /billing/callback`، و`metadata = { tenant_id, plan_code, cycle }`.
- **الإخراج:** `{ publishableKey, amount: priceMinor, currency: 'SAR', callbackUrl, metadata }`. الواجهة تُقدّم نموذج Moyasar.js (mada/Apple Pay/بطاقة).
- بطاقة البيانات لا تمرّ بنا أبداً.

### `POST /billing/webhook`
المُستقبِل الموثوق لتأكيد Moyasar — **هو ما يفعّل الاشتراك فعلياً**.
- عام (لا Auth tenant)، يُستدعى خادم-لخادم من Moyasar.
- **التحقّق (طبقتان):**
  1. مقارنة `secret_token` بالسرّ المخزّن عبر `timingSafeEqual` → عدم تطابق = `401` وإسقاط.
  2. `GET /v1/payments/{data.id}` بـ`sk_*` ومقارنة الحالة الفعلية (يكشف الـreplay والتزوير).
- **عند `payment_paid` متحقّق:** (idempotent على `payment.id`) → تحويل `Subscription.status` إلى `active`، ضبط `currentPeriodEnd`، إصدار `Invoice` متوافق ZATCA، إعادة ضبط نافذة الاستهلاك الشهرية.
- **عند `payment_failed`:** `trialing`/`active` → `past_due` (وصول معلّق) + رسالة واضحة للعميل (US-1.2).
- يردّ `200` بسرعة (يُدرج التفعيل في طابور إن لزم)؛ لا يردّ `5xx` لـ"عولج مسبقاً" بل `200`.

### `GET /billing/subscription`
يعرض حالة الاشتراك الحالي للعميل (US-1.3).
- **الإخراج:** `{ status, planCode, priceSar, cycle, trialEndsAt?, currentPeriodEnd?, usage: { drafts: { used, cap }, images: { used, cap } } }`.
- `usage` يُحسب من `UsageRecord` للنافذة الحالية مقابل سقوف الباقة.

### `POST /billing/cancel`
يُلغي الاشتراك (US-1.3).
- يضبط `Subscription.status = 'canceled'` ويُعلّم `cancelAtPeriodEnd` (الوصول يبقى حتى نهاية الفترة المدفوعة، ثم يُعلّق).
- لا استرداد تلقائي عند الإطلاق (يدوي عبر لوحة Moyasar إن لزم).

### `GET /billing/invoice/:id`
يحمّل فاتورة ZATCA (US-1.3).
- يتحقّق أن `invoice.tenantId === ctx.tenantId` (عزل) → `404` إن لا.
- يعيد بيانات الفاتورة (أو PDF متوافق ZATCA مع QR).

### `BillingService` (داخلي)
المنطق المشترك خلف المسارات: `createPaymentIntent`, `verifyAndActivate`, `issueZatcaInvoice`, `transitionStatus`.

### `UsageGuard` (يستهلكه المحرّك — المرحلة ١)
```ts
canConsume(tenantId: string, kind: UsageKind): Promise<ConsumeDecision>

interface ConsumeDecision {
  allowed: boolean
  used: number
  cap: number
  reason?: string   // رسالة عربية عند الرفض: "بلغت سقف الباقة الشهري للمسودّات"
}
```
- يقرأ مجموع `UsageRecord` (لنفس `tenantId` + `kind`) داخل النافذة الشهرية مقابل سقف الباقة: `monthlyDraftCap` لـ`text`، `monthlyImageCap` لـ`image`، `monthlySearchCap` لـ`search`. **كل قيم `UsageKind` الثلاث لها سقف ويُفحص** — البحث استدعاء مكلف فلا يُترك بلا حدّ (NFR-7).
- `allowed=false` عند التجاوز → المحرّك يرفض التوليد قبل أي استدعاء AI ويُظهر `reason`.
- اشتراك `past_due`/`canceled` → `allowed=false` بصرف النظر عن العداد.

## التدفّقات

### تدفّق ١ — الاشتراك الأول (نهاية التجربة، US-1.2)
```
العميل (trialing) ─▶ POST /billing/subscribe
   └─▶ النظام: PlanDefinition + given_id + metadata{tenant_id} ─▶ بيانات Moyasar.js للواجهة
العميل: يُدخل mada/Apple Pay/بطاقة في نموذج Moyasar (لا يمرّ بنا)
Moyasar: ينشئ الدفعة
   ├─ status='paid'      → redirect callback_url
   └─ status='initiated' → العميل يكمل تحدّي 3DS على source.transaction_url ثم callback
[القناة الموثوقة] Moyasar ─▶ POST /billing/webhook (payment_paid)
   1) timingSafeEqual(secret_token)   2) GET /v1/payments/:id (sk_*)
   3) تحقّق status='paid' + amount + currency + metadata.tenant_id
   4) Subscription.status: trialing → active ; إصدار Invoice (ZATCA) ; إعادة ضبط نافذة الاستهلاك
[UX] callback ─▶ صفحة نجاح (تأكيد فقط، لا تفعيل)
```

### تدفّق ٢ — الاستهلاك بعد التفعيل (يُستدعى من المحرّك)
```
المحرّك (قبل توليد بوست/صورة/بحث) ─▶ UsageGuard.canConsume(tenantId, 'text'|'image'|'search')
   ├─ allowed=true  → يكمل التوليد ─▶ يُسجّل UsageRecord(kind, units, costUsd)
   └─ allowed=false → رفض + رسالة "بلغت سقف الباقة الشهري" (لا استدعاء AI)
```

### تدفّق ٣ — فشل الدفع / التجديد (US-1.2)
```
Moyasar ─▶ POST /billing/webhook (payment_failed)
   └─▶ Subscription.status → past_due ؛ الوصول معلّق ؛ رسالة واضحة + دعوة لإعادة المحاولة
العميل يعيد المحاولة ─▶ POST /billing/subscribe ─▶ (دفعة ناجحة) ─▶ past_due → active
```

### تدفّق ٤ — العرض/الإلغاء (US-1.3)
```
GET /billing/subscription  → الحالة + الاستهلاك مقابل السقوف
POST /billing/cancel        → canceled (cancelAtPeriodEnd) ، الوصول حتى نهاية الفترة
GET /billing/invoice/:id    → تحميل فاتورة ZATCA (تحقّق tenant)
```

## ZATCA — متطلبات الفوترة الإلكترونية

التزام **المرحلة الأولى (Generation)** كحدّ أدنى عند الإطلاق؛ بنية الـ`Invoice` تستوعب المرحلة الثانية (Integration/Clearance) لاحقاً. الوصف هنا متطلّبات، لا تنفيذ تفصيلي.

- **الحقول الإلزامية:** اسم البائع (أثر) ورقمه الضريبي (`sellerVatNumber`)، اسم المشتري، تاريخ/وقت الإصدار، رقم فاتورة تسلسلي، الإجمالي قبل الضريبة، نسبة ومبلغ ضريبة القيمة المضافة (١٥٪)، الإجمالي شامل الضريبة، العملة `SAR`.
- **رمز QR متوافق:** ترميز **TLV** ثم Base64 يتضمّن (1) اسم البائع، (2) الرقم الضريبي، (3) الطابع الزمني، (4) الإجمالي شامل الضريبة، (5) مبلغ الضريبة. يُخزَّن في `Invoice.zatcaQr` ويُعرض على الفاتورة.
- **فاتورة ضريبية مبسّطة (B2C)** هي الافتراض لاشتراك SMB؛ تُرفّع لفاتورة ضريبية كاملة إن قدّم العميل رقمه الضريبي.
- **الترقيم التسلسلي** فريد ومتّصل لكل `tenant` (لا فجوات).
- **PDPL:** تقليل البيانات في الفاتورة، والاحتفاظ بها للمدة النظامية، ودعم حق الحذف بعدها (NFR-5).
- لا تُخزَّن أي بيانات بطاقة في الفاتورة أو السجلّات (آخر ٤ أرقام فقط إن لزم العرض).

## معالجة الأخطاء

| الموضع | الفشل | السلوك |
|---|---|---|
| دفع أول | `payment_failed` | `trialing/active → past_due`؛ وصول معلّق؛ رسالة واضحة + إعادة محاولة (US-1.2) |
| دفع | عالق على `initiated` (3DS غير مكتمل) | لا تفعيل؛ العميل يُوجّه لـ`source.transaction_url`؛ يُعاد ضبطه إن لم يكتمل |
| webhook | `secret_token` غير مطابق (مزوّر) | `401` وإسقاط فوري قبل أي معالجة |
| webhook | حدث مكرّر/replay | idempotent على `payment.id`؛ re-fetch يكشف الحالة الفعلية؛ يردّ `200` بلا أثر جانبي |
| webhook | تباين amount/currency/metadata عن المتوقّع | لا تفعيل؛ تسجيل وتنبيه؛ يُعامل كمحاولة تلاعب |
| تجربة | انتهت بلا دفعة | `trialing → past_due`؛ وصول معلّق؛ دعوة للاشتراك (US-1.2) |
| استهلاك | تجاوز سقف الباقة | `canConsume=false`؛ رفض التوليد قبل استدعاء AI؛ رسالة عربية واضحة |
| استهلاك | بلوغ السقف وسط توليد خطة شهر (٣٠ بوست) | البوستات المتبقية تُعلَّم `skipped_quota` (حالة مختلفة عن فشل المزوّد)؛ يُبلَّغ العميل بعدد المتبقّي وسبب التوقّف؛ **بلا إسقاط الخطة** — ما اكتمل يبقى |
| استهلاك | اشتراك `past_due/canceled` | رفض كل توليد بصرف النظر عن العداد |
| اشتراك مزدوج | `given_id` مكرّر (`400 already created`) | يُعامل كـ"الدفعة موجودة، اجلبها"، لا كفشل |

## الاعتمادات

- **Moyasar API** (`api.moyasar.com/v1`) — دفعات، توكنات، فواتير؛ Basic Auth بالمفتاح السرّي.
- **Moyasar.js** (الواجهة — المرحلة ٧) — نموذج mada / Apple Pay / بطاقات (المفتاح القابل للنشر فقط).
- **نموذج Prisma الموجود:** `Subscription`, `UsageRecord`, `Tenant` — كلها معرّفة في خطة الأساس (Sprint 0). حقول `currentPeriodEnd`, `cancelAtPeriodEnd`, `trialEndsAt` و enum `SubscriptionStatus` **موجودة أصلاً في الأساس**؛ هذه المرحلة تستهلكها ولا تُوسّع `Subscription` بأي حقل. العلاقة `Tenant.subscriptions Subscription[]` (واحد-لمتعدّد، تاريخ متعدّد) — معرّفة في الأساس. **الجدول الجديد لهذه المرحلة = `Invoice` فقط** (لا يوجد بعد)، يُضاف بـmigration مستقل.
- **المرحلة ٣ (Auth):** تنشئ سجلّ `Subscription` بـ`trialing` للـ`tenant` عند التسجيل وتوفّر سياق `tenantId`. العلاقة تاريخية (`subscriptions Subscription[]`): عند دورة `past_due ↔ active` قد يتراكم أكثر من سجلّ اشتراك لنفس الـ`tenant`، فالاشتراك «الحالي» هو الأحدث. هنا فقط منطق الاشتراك، لا الـAuth.
- **المرحلة ١ (المحرّك):** يستهلك `UsageGuard.canConsume(...)` قبل كل توليد.
- **سرّ webhook المشترك** + المفتاح السرّي — في secrets manager، لا في git.
- **مكتبة ZATCA QR (TLV/Base64)** لتوليد رمز الفاتورة.

## خارج النطاق

- **الواجهة (frontend):** نموذج الدفع، صفحات الاشتراك/الفاتورة، رسائل الحالة — المرحلة ٧.
- **الـAuth والتسجيل:** المرحلة ٣ — هنا فقط منطق الاشتراك بعد وجود `tenantId` و`Subscription` مبدئي.
- **منطق المحرّك:** المرحلة ١ — هذه المرحلة توفّر `canConsume` فقط، لا التوليد.
- **باقات متعدّدة / ترقية-تخفيض / proration** — باقة واحدة عند الإطلاق؛ مؤجّل.
- **الاسترداد التلقائي (refund flow)** — يدوي عبر لوحة Moyasar عند الإطلاق.
- **ZATCA المرحلة الثانية (Integration/Clearance مع منصّة فاتورة)** — البنية تستوعبها، التكامل مؤجّل.
- **التحصيل الآلي للتجديد المتكرّر بالتوكن (saved-card recurring)** — يُقيَّم بعد الإطلاق.

## معايير القبول

- عميل في `trialing` يُتمّ دفعة Moyasar ناجحة (mada/Apple Pay/بطاقة) → `webhook` يحوّله إلى `active` ويصدر فاتورة ZATCA (US-1.2).
- فشل الدفع يُبقي الوصول معلّقاً (`past_due`) مع رسالة عربية واضحة، وإعادة المحاولة الناجحة تُعيده `active` (US-1.2).
- اعتماد الاشتراك يتمّ **حصراً** بعد `GET /v1/payments/:id` بالمفتاح السرّي ومطابقة `status`/`amount`/`currency`/`metadata.tenant_id` — لا اعتماد من الـ`callback`.
- `webhook` مكرّر/replay لا يُنتج تفعيلاً مزدوجاً (idempotent على `payment.id`)، و`secret_token` غير المطابق يُرفض `401`.
- `canConsume(tenantId, 'text'|'image'|'search')` يرفض التوليد عند تجاوز السقف الشهري لأيٍّ من الأنواع الثلاثة (بما فيها `search` — NFR-7) أو عند `past_due/canceled`، ويُظهر رسالة واضحة، قبل أي استدعاء AI.
- انتهاء التجربة بلا دفع يُحوّل `trialing → past_due` ويُعلّق الوصول.
- `GET /billing/subscription` يعرض الحالة والاستهلاك مقابل السقوف؛ `POST /billing/cancel` يُلغي مع إبقاء الوصول حتى نهاية الفترة؛ `GET /billing/invoice/:id` يحمّل فاتورة ZATCA مقيّدة بـ`tenant`.
- المبلغ يُمرَّر بالهللة (عدد صحيح) في كل المسارات؛ لا بيانات بطاقة تلمس الخادم.
- كل سجلّ (اشتراك/استهلاك/فاتورة) معزول بـ`tenantId`؛ لا تسرّب بين العملاء (NFR-5).
