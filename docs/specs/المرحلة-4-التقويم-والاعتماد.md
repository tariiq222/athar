# المرحلة ٤ — التقويم والاعتماد (Calendar & Approval)

> Spec تصميمي لمرحلة تنظيم المحتوى واعتماده قبل النشر في منصّة **أثر**.
> النطاق: `FR-8`, `FR-9`, `FR-10` — القصص: `US-4.1`, `US-4.2`.
> الكود إنجليزي فقط. الشرح هنا بالعربي، والأنواع والمسارات بالإنجليزية.

---

## 1. نظرة عامة (Overview)

هذه المرحلة هي **حلقة الوصل بين توليد المحتوى (المرحلة ١) والنشر (المرحلة ٥)**. المحرّك في المرحلة ١ ينتج كائنات `Post` بحالة `draft`؛ مهمّة هذه المرحلة أن تَعرض هذه المسودّات في **تقويم محتوى** يقترح مواعيد ومناسبات سعودية، وأن تُدير **سير حالاتها** (`draft → pending_review → approved`) مع السماح بالتعديل قبل الاعتماد، وأن تُعيّن **موعد تذكير نشر** (`scheduledAt`) لكل بوست.

ثلاث قدرات أساسية:

1. **تقويم محتوى (FR-8):** عرض المناسبات السعودية (وطنية، تأسيس، رمضان، عيدان، مواسم تجارية) إلى جانب البوستات المجدولة لفترة زمنية، لمساعدة المستخدم على اختيار توقيت يلائم المناسبة.
2. **سير اعتماد (FR-9):** آلة حالة صارمة تنقل البوست عبر `draft → pending_review → approved`، مع تعديل النص/الهاشتاق/الصورة قبل الاعتماد، ورفض أي انتقال غير مسموح.
3. **جدولة تذكير نشر (FR-10):** تعيين `scheduledAt` للبوست في الموعد المقترح. **تنفيذ التذكير نفسه خارج النطاق — يُنفَّذ في المرحلة ٥.** هنا فقط نُخزّن الموعد.

**ما هذه المرحلة ليست عنه:** لا توليد محتوى (المرحلة ١)، لا نشر فعلي ولا تنفيذ تذكير (المرحلة ٥)، لا واجهة مستخدم (المرحلة ٧). **النشر يدوي ولا يوجد API نشر في هذه المرحلة.**

## 2. المبادئ (Principles)

- **Multi-tenant منطقي:** كل صفّ يحمل `tenantId`. كل استعلام وكل تحوّل حالة **يجب** أن يُقيَّد بـ `tenantId` للمستأجر الحالي. لا يُسمح بقراءة أو تعديل بوست خارج المستأجر.
- **آلة حالة صريحة لا ضمنية:** الانتقالات بين الحالات تمرّ عبر جدول انتقالات مُعرّف (`PostStatusTransition`). أي انتقال غير موجود في الجدول يُرفض بخطأ `INVALID_TRANSITION`. لا تُحدَّث `status` مباشرة بأي مسار آخر.
- **البناء على النموذج الموجود:** نوسّع `Post` و `ImageAsset` و `SourceCitation` الموجودة في Prisma، ولا نُعيد تصميمها. المناسبات السعودية كيان جديد مستقل (`SaudiOccasion`).
- **المناسبات بيانات مرجعية (seed)، لا منطق محسوب وقت الطلب:** التواريخ الهجرية تُحسب مسبقاً وتُخزَّن كصفوف ميلادية في جدول `SaudiOccasion` لكل سنة. لا حساب هجري داخل مسار الطلب.
- **القراءة فصلٌ عن الكتابة:** `GET /calendar` و `GET /posts` للقراءة فقط؛ التعديل وتحويل الحالة عبر `PATCH /posts/:id` حصراً.
- **عدم تغيير لا رجعة فيه ضمنياً:** الانتقال إلى `approved` لا يحذف القدرة على القراءة، لكن لا يُسمح بالعودة منه تلقائياً (انظر آلة الحالة). `approved → published` يحدث في المرحلة ٥ فقط.
- **الأخطاء صريحة ومُصنَّفة:** كل خطأ يحمل رمزاً ثابتاً (`code`) قابلاً للترجمة في طبقة الواجهة لاحقاً.

## 3. الأنواع (Types)

> أنواع TypeScript (NestJS DTO/domain). أسماء الحقول إنجليزية. التواريخ ISO-8601 (UTC).

```ts
// نوع المناسبة السعودية — صفّ مرجعي مُولَّد بالـ seed لكل سنة ميلادية
type SaudiOccasionKind =
  | 'national'        // اليوم الوطني
  | 'foundation'      // يوم التأسيس
  | 'ramadan'         // شهر رمضان (نطاق)
  | 'eid_fitr'        // عيد الفطر
  | 'eid_adha'        // عيد الأضحى
  | 'commercial';     // موسم تجاري (عروض، تخفيضات، عودة المدارس...)

interface SaudiOccasion {
  id: string;
  tenantId: string | null;   // null = مناسبة عامة لكل المستأجرين؛ قيمة = مناسبة خاصة بمستأجر
  slug: string;              // مُعرّف ثابت للمناسبة (مفتاح الـ upsert) — مثل 'saudi-national-day', 'back-to-school', 'white-friday'
  kind: SaudiOccasionKind;
  nameAr: string;            // اسم العرض بالعربي (محتوى بيانات، لا كود)
  nameEn: string;
  startDate: string;         // ISO date (ميلادي محسوب من الهجري وقت الـ seed)
  endDate: string;           // ISO date — يساوي startDate للمناسبة ذات اليوم الواحد
  hijriYear: number;         // السنة الهجرية المرجعية المستخدمة في الحساب
  gregorianYear: number;     // السنة الميلادية التي يقع فيها الصفّ
}

// إدخال موحّد في التقويم: إمّا مناسبة وإمّا بوست مجدول
type CalendarEntryType = 'occasion' | 'post';

interface CalendarEntry {
  type: CalendarEntryType;
  date: string;                 // ISO date — اليوم الذي يظهر فيه الإدخال
  occasion?: SaudiOccasion;     // مُعبَّأ عندما type === 'occasion'
  post?: CalendarPostSummary;   // مُعبَّأ عندما type === 'post'
}

// ملخّص بوست داخل التقويم (لا نُعيد النص الكامل في عرض التقويم)
interface CalendarPostSummary {
  id: string;
  platform: 'linkedin' | 'x';
  status: 'draft' | 'pending_review' | 'approved' | 'published';
  scheduledAt: string | null;
  excerpt: string;              // أول ~120 حرف من النص
  hasImage: boolean;
}

// انتقال حالة مطلوب عبر PATCH
type PostStatus = 'draft' | 'pending_review' | 'approved' | 'published';

interface PostStatusTransition {
  from: PostStatus;
  to: PostStatus;
}

// قائمة الانتقالات المسموحة في هذه المرحلة (مرجع آلة الحالة)
const ALLOWED_TRANSITIONS: readonly PostStatusTransition[] = [
  { from: 'draft',          to: 'pending_review' },
  { from: 'pending_review', to: 'approved' },
  { from: 'pending_review', to: 'draft' },     // إعادة للمسودّة لمزيد من التعديل
  { from: 'approved',       to: 'pending_review' }, // سحب الاعتماد قبل النشر
  // { from: 'approved',     to: 'published' }  // المرحلة ٥ فقط — ليس هنا
];
```

### الامتداد على نموذج Prisma الموجود

لا تعديل على بنية `Post` / `ImageAsset` / `SourceCitation` القائمة. الإضافة الوحيدة جدول جديد:

```prisma
model SaudiOccasion {
  id             String   @id @default(cuid())
  tenantId       String?  // null = عام لكل المستأجرين
  slug           String   // مُعرّف ثابت للمناسبة — مفتاح الـ upsert
  kind           String   // SaudiOccasionKind
  nameAr         String
  nameEn         String
  startDate      DateTime
  endDate        DateTime
  hijriYear      Int
  gregorianYear  Int

  // مفتاح الـ upsert: مناسبة واحدة لكل (مستأجر، slug، سنة ميلادية).
  // يسمح بعدّة صفوف commercial في نفس السنة لأنّ لكلٍّ slug مختلف
  // ('back-to-school', 'white-friday', 'year-end-sale'...).
  @@unique([tenantId, slug, gregorianYear])
  @@index([tenantId, startDate, endDate])
  @@index([gregorianYear])
}
```

> ملاحظة: `Post.scheduledAt` و `Post.status` موجودان مسبقاً — هذه المرحلة تكتب فيهما فقط، لا تُنشئهما.

## 4. المكوّنات وواجهات REST (Components & REST Interfaces)

### 4.1 المكوّنات (NestJS modules)

- **`CalendarModule`** — يجمّع المناسبات والبوستات المجدولة في `CalendarEntry[]` لفترة. يعتمد على `OccasionService` و `PostService` (للقراءة).
- **`OccasionModule` / `OccasionService`** — قراءة المناسبات السعودية (عامّة + خاصّة بالمستأجر) ضمن نطاق تواريخ. مصدرها جدول `SaudiOccasion` (seed).
- **`PostModule` / `PostService`** — قراءة البوستات بفلترة، وتطبيق `PATCH` (تعديل + تحويل حالة عبر آلة الحالة).
- **`PostStateMachine`** — مكوّن نقيّ (pure) يتحقّق من `ALLOWED_TRANSITIONS` ويُرجع قبول/رفض الانتقال. لا وصول لقاعدة البيانات.
- **`HijriOccasionSeeder`** — أمر/سكربت seed يحسب التواريخ الهجرية ويكتب صفوف `SaudiOccasion` لسنة ميلادية. ليس مساراً HTTP.

### 4.2 `GET /calendar`

يُرجع المناسبات + البوستات المجدولة ضمن فترة، مدمجة كـ `CalendarEntry[]`.

| البند | القيمة |
|---|---|
| Query | `from` (ISO date, مطلوب), `to` (ISO date, مطلوب), `platform?` (`linkedin`\|`x`), `kind?` (`SaudiOccasionKind`) |
| القيد | `tenantId` من السياق دائماً. المدى الأقصى للفترة: ٩٢ يوماً (يُرفض الأطول بـ `RANGE_TOO_WIDE`). |
| الإرجاع | `{ entries: CalendarEntry[] }` مرتّبة تصاعدياً بالتاريخ |
| المناسبات | عامّة (`tenantId = null`) + خاصّة بالمستأجر، ضمن `[from, to]` |
| البوستات | البوستات التي لها `scheduledAt` ضمن `[from, to]` لهذا المستأجر |

### 4.3 `GET /posts`

قائمة البوستات بفلترة، للقراءة (إدارة قائمة الاعتماد).

| البند | القيمة |
|---|---|
| Query | `status?` (`PostStatus`), `platform?` (`linkedin`\|`x`), `from?` / `to?` (نطاق `scheduledAt`), `page?`, `pageSize?` (افتراضي ٢٠، أقصى ١٠٠) |
| القيد | `tenantId` من السياق دائماً |
| الإرجاع | `{ items: PostListItem[], page, pageSize, total }` |
| الترتيب | `scheduledAt` تصاعدياً ثم `createdAt` تنازلياً (البوستات بلا موعد في النهاية) |

`PostListItem` يحمل: `id, platform, status, scheduledAt, text, hashtags, hasImage, citationCount`.

### 4.4 `PATCH /posts/:id`

المسار الوحيد للكتابة: تعديل المحتوى و/أو تحويل الحالة. **العمليتان ذرّيّتان معاً** (transaction واحدة).

| البند | القيمة |
|---|---|
| Body | `{ text?, hashtags?, image?: { url, method } | null, scheduledAt?: string | null, transition?: PostStatusTransition }` |
| القيد | `tenantId` من السياق؛ يُرفض إن لم يكن البوست للمستأجر بـ `NOT_FOUND` (لا تسريب وجود) |
| التعديل | يُسمح بتعديل `text` / `hashtags` / `image` فقط حين الحالة `draft` أو `pending_review`. لا تعديل محتوى لبوست `approved` (يجب سحبه أولاً إلى `pending_review`). |
| الموعد | `scheduledAt: <ISO datetime>` يعيّن موعد التذكير المقترح (يجعل البوست يظهر في `GET /calendar` كـ `type='post'`)؛ `scheduledAt: null` يمسح الموعد. لا يقيّده تحويل الحالة. |
| الحالة | إن وُجد `transition`، يجب أن يطابق `transition.from` الحالة الحالية الفعلية، وأن يكون ضمن `ALLOWED_TRANSITIONS`، وإلّا `INVALID_TRANSITION` |
| الصورة | `image: null` يحذف `ImageAsset` المرتبط؛ `image: {url, method}` يُنشئ/يُحدّث (علاقة `postId` فريدة) |
| الإرجاع | البوست بعد التحديث (`PostDetail` يشمل الصورة والاستشهادات) |

> قاعدة الترتيب داخل الطلب: يُطبَّق تعديل المحتوى أولاً، ثم يُتحقَّق من الانتقال على الحالة القائمة، ثم يُكتب الانتقال — كله في transaction واحدة. فشل أي خطوة يُرجِع كامل الطلب دون أثر.

## 5. التدفّقات (Flows)

### 5.1 تدفّق الاعتماد (US-4.2)

```
المحرّك (مرحلة ١) ─▶ Post(status=draft)
                          │
   GET /posts?status=draft ─▶ المراجِع يستعرض المسودّات
                          │
   PATCH /posts/:id { text?, hashtags?, image? }      ← تعديل قبل المراجعة (اختياري)
                          │
   PATCH /posts/:id { transition: draft→pending_review }
                          │
   PATCH /posts/:id { transition: pending_review→approved }   ← اعتماد
                          │
                   Post(status=approved)
                          │
        (المرحلة ٥) approved→published — خارج نطاق هذه المرحلة
```

مسارات فرعية:
- **إرجاع للتعديل:** `pending_review → draft` لإعادة فتح التعديل.
- **سحب الاعتماد:** `approved → pending_review` قبل النشر (طالما لم يُنشر بعد).

### 5.2 تدفّق التقويم وجدولة التذكير (US-4.1, FR-8, FR-10)

```
GET /calendar?from&to ─▶ entries: [occasions + scheduled posts]
                          │
   المستخدم يختار يوماً قرب مناسبة (مثلاً اليوم الوطني)
                          │
   PATCH /posts/:id { /* تعيين الموعد */ scheduledAt ضمن التعديل }
                          │
            Post.scheduledAt = <الموعد المقترح>
                          │
   يظهر البوست في GET /calendar كـ CalendarEntry(type='post')
                          │
        (المرحلة ٥) منفّذ التذكير يقرأ scheduledAt ويُذكّر — خارج النطاق هنا
```

> `scheduledAt` حقل موجود في `Post`. تعيينه عبر `PATCH /posts/:id` ضمن body التعديل. تنفيذ التذكير (cron/queue) **ليس** في هذه المرحلة.

### 5.3 حساب التواريخ الهجرية وتحديثها السنوي

- **المصدر:** جدول `SaudiOccasion` المُولَّد بالـ seed، لا حساب وقت الطلب.
- **الحساب:** يستخدم `HijriOccasionSeeder` مكتبة هجرية (مثل `hijri-converter` / `moment-hijri`) لتحويل التواريخ الهجرية الثابتة (١ رمضان، ١ شوّال = عيد الفطر، ١٠ ذو الحجة = عيد الأضحى) إلى تواريخ ميلادية لسنة ميلادية مستهدفة، ويكتب صفّاً لكل مناسبة.
- **الثوابت الميلادية:** اليوم الوطني (٢٣ سبتمبر) ويوم التأسيس (٢٢ فبراير) تواريخ ميلادية ثابتة — تُكتب مباشرة دون تحويل هجري.
- **المواسم التجارية:** نطاقات ميلادية ثابتة تقريبية (عودة المدارس، الجمعة البيضاء/نوفمبر، تخفيضات نهاية العام) كصفوف `kind='commercial'`.
- **التحديث السنوي:** يُشغَّل الـ seeder سنوياً لتوليد صفوف السنة الميلادية القادمة. كل صفّ يحمل `slug` ثابتاً و `gregorianYear` و `hijriYear` لتتبّع المصدر ومنع التكرار. التشغيل يدوي/مجدول إدارياً (لا يخصّ مسار المستخدم). إعادة التشغيل لنفس `gregorianYear` تكون idempotent عبر upsert على المفتاح الفريد `(tenantId, slug, gregorianYear)` (للمناسبات العامّة يكون `tenantId = null`). هذا المفتاح يسمح بعدّة مناسبات `kind='commercial'` في نفس السنة لأنّ لكلٍّ `slug` مميّز (`back-to-school`، `white-friday`، `year-end-sale`...)، بخلاف مفتاح `(tenantId, kind, gregorianYear)` الذي كان سيدمج هذه المواسم في صفّ واحد.
- **الدقّة:** بدايات الأشهر الهجرية قد تختلف يوماً برؤية الهلال؛ التواريخ المُولَّدة **تقديرية للتخطيط** ويُسمح بتصحيحها يدوياً في صفوف `SaudiOccasion`.

## 6. الأخطاء (Errors)

كل خطأ يُرجَع بـ HTTP status مناسب + جسم `{ code, message }` (الرسالة قابلة للترجمة لاحقاً).

| `code` | HTTP | المعنى |
|---|---|---|
| `VALIDATION_FAILED` | 400 | جسم/استعلام غير صالح (تاريخ غير ISO، حقل ناقص) |
| `RANGE_TOO_WIDE` | 400 | فترة التقويم تتجاوز ٩٢ يوماً |
| `INVALID_TRANSITION` | 409 | انتقال غير مسموح أو `from` لا يطابق الحالة الحالية |
| `CONTENT_LOCKED` | 409 | محاولة تعديل محتوى بوست `approved` (يجب سحبه أولاً) |
| `NOT_FOUND` | 404 | البوست غير موجود أو خارج المستأجر (لا تمييز — منعاً للتسريب) |
| `PUBLISH_NOT_ALLOWED_HERE` | 422 | محاولة `approved→published` في هذه المرحلة (يُنفَّذ في المرحلة ٥) |

قواعد:
- خرق `tenantId` (وصول لبوست مستأجر آخر) يُعامَل دائماً كـ `NOT_FOUND` لا `FORBIDDEN`.
- التحقّق من الانتقال يسبق أي كتابة؛ فشله لا يترك أثراً جزئياً.

## 7. الاعتمادات (Dependencies)

- **المرحلة ١ — المحرّك:** يُنتج `Post(status=draft)`. هذه المرحلة **مستهلِكة** لها فقط؛ لا تُولّد بوستات. لا تعديل على عقد المحرّك.
- **نموذج Prisma الموجود:** `Post`, `ImageAsset`, `SourceCitation`, `BrandProfile` — تُقرأ/تُحدَّث كما هي. الإضافة الوحيدة جدول `SaudiOccasion`.
- **سياق المستأجر:** يعتمد على middleware/guard يوفّر `tenantId` (مفترض موجود من البنية العامّة). كل خدمة تأخذ `tenantId` صراحةً.
- **مكتبة هجرية:** `hijri-converter` أو ما يماثلها — يستخدمها الـ seeder فقط، خارج مسار الطلب.
- **المرحلة ٥ — النشر والتذكير:** مستهلِكة لمخرجات هذه المرحلة (`scheduledAt`, `status=approved`). تملك انتقال `approved→published` وتنفيذ التذكير.

## 8. خارج النطاق (Out of Scope)

- **توليد المحتوى** (المرحلة ١): إنشاء نصّ البوست، الصور، الاستشهادات.
- **النشر الفعلي** (المرحلة ٥): لا API نشر. النشر يدوي. انتقال `approved→published` يحدث في المرحلة ٥.
- **تنفيذ التذكير** (المرحلة ٥): جدولة/تشغيل cron أو queue للتذكير. هنا فقط تعيين `scheduledAt`.
- **واجهة المستخدم** (المرحلة ٧): لا UI، لا مكوّنات عرض تقويم.
- **حساب هجري وقت الطلب:** كل التواريخ مُولَّدة مسبقاً بالـ seed.
- **إدارة المستأجرين والصلاحيات:** خارج هذه المرحلة (مفترض من البنية العامّة).

## 9. معايير القبول (Acceptance Criteria)

- **AC-1 (FR-8):** `GET /calendar?from&to` يُرجع المناسبات السعودية العامّة + الخاصّة بالمستأجر **والبوستات المجدولة** ضمن الفترة، مدمجة كـ `CalendarEntry[]` مرتّبة بالتاريخ، ومقيّدة بـ `tenantId`.
- **AC-2 (FR-8):** المناسبات تشمل الأنواع الستّة (`national`, `foundation`, `ramadan`, `eid_fitr`, `eid_adha`, `commercial`) لسنة ميلادية مُولَّدة بالـ seed، مع تواريخ هجرية محسوبة وقابلة للتحديث السنوي بشكل idempotent.
- **AC-3 (FR-9):** `PATCH /posts/:id` ينقل الحالة عبر `ALLOWED_TRANSITIONS` فقط؛ أي انتقال غير مسموح أو `from` غير مطابق يُرفض بـ `INVALID_TRANSITION` (409) دون تغيير الحالة.
- **AC-4 (FR-9):** يُسمح بتعديل `text`/`hashtags`/`image` فقط حين `draft` أو `pending_review`؛ تعديل بوست `approved` يُرفض بـ `CONTENT_LOCKED`.
- **AC-5 (FR-9):** التعديل وتحويل الحالة في `PATCH` واحد يجريان في transaction واحدة؛ فشل أيّهما لا يترك أثراً جزئياً.
- **AC-6 (FR-10):** تعيين `scheduledAt` عبر `PATCH /posts/:id` يجعل البوست يظهر في `GET /calendar` كـ `CalendarEntry(type='post')` ضمن فترته. لا يوجد تنفيذ تذكير في هذه المرحلة.
- **AC-7 (multi-tenant):** كل قراءة/كتابة مقيّدة بـ `tenantId`؛ الوصول لبوست مستأجر آخر يُرجع `NOT_FOUND`.
- **AC-8 (المحرّك):** البوستات المُنتَجة بحالة `draft` من المرحلة ١ تَظهر في `GET /posts?status=draft` وتقبل سير الاعتماد دون أي تعديل على المحرّك.
- **AC-9 (حدود):** فترة تقويم أطول من ٩٢ يوماً تُرفض بـ `RANGE_TOO_WIDE`؛ محاولة `approved→published` تُرفض بـ `PUBLISH_NOT_ALLOWED_HERE`.
