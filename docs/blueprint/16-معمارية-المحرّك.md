# معمارية محرّك المحتوى — أثر (Spec)

> القلب والخطر في المنتج. خط أنابيب هجين: مراحل بحث/مصادر منفصلة (تحكّم في الاستشهاد) ثم كتابة Claude الحرة من الحقائق، مع نقد ذاتي يرفع الجودة قبل عين العميل. مدفوع بمدخلات العميل (`BrandProfile`). يطبّق حدود المنصات من [15-مواصفات-المنصات.md](15-مواصفات-المنصات.md).
>
> أسماء الأنواع والواجهات بالإنجليزية (code-facing). الشرح بالعربي.

## المبادئ
- **تحكّم في الاستشهاد:** المصادر تُجمع في مرحلة مخصّصة؛ Claude يكتب من حقائق مُتحقّقة لا من فراغ. لا اختلاق مصدر أبداً.
- **خط واضح قابل للاختبار:** كل مرحلة وحدة مستقلة بواجهة واضحة، تُختبر وتُشخّص وحدها.
- **الجودة قبل العميل:** حلقة نقد ذاتي ترفع المسودّة قبل العرض — هذا الفرق عن GPT خام.
- **مدفوع بمدخلات العميل:** كل مرحلة تقرأ `BrandProfile` (نبرة، محاور، ممنوعات، جمهور، أهداف، تفضيلات متعلّمة).
- **التكلفة كقيد:** عدد استدعاءات البحث/التوليد محدود لكل بوست؛ كل استدعاء يُسجّل في `UsageRecord`.
- **المزوّد خلف interface:** التوليد النصّي خلف `ContentProvider` فقط (تبديل/اختبار نموذج لاحقاً).

## المدخلات (أنواع البيانات)

```ts
type Platform = 'linkedin' | 'x'
type ContentType = 'informational' | 'thought' | 'announcement' | 'engagement'

interface BrandProfile {
  id: string
  tenantId: string
  tone: string            // وصف النبرة
  topics: string[]        // محاور العميل (setup + اقتراحات النظام)
  audience: string
  goals: string
  prohibitions: string[]  // ممنوعات/محظورات
  competitors: string[]
  keywords: string[]
  brandKit: BrandKit      // الهوية البصرية للصور
  learnedPreferences: string  // سياق متراكم من تعديلات العميل (تعلّم خفيف)
}

interface BrandKit {
  colors: string[]
  logoUrl?: string
  visualStyle: string     // وصف الستايل لحقن gpt-image
  font: string            // افتراضي IBM Plex Sans Arabic (لطبقة overlay الاحتياطية)
}

interface GenerationRequest {
  brandProfile: BrandProfile
  platform: Platform
  contentType: ContentType
  brief?: string          // توجيه اختياري للبوست (مرونة العميل)
  topic?: string          // محور محدّد، وإلا يُختار من brandProfile.topics
}
```

## المراحل

### المرحلة ١ — بحث ومصادر (مُتحكَّم بها)
```ts
research(topic: string, brand: BrandProfile): Promise<FactSet>

interface FactSet {
  hasFactualClaim: boolean
  facts: Fact[]           // فارغ إن لم يوجد مصدر موثوق
}
interface Fact {
  claim: string           // المعلومة كما وردت في المصدر
  sourceUrl: string
  sourceTitle: string
  confidence: number      // 0..1
}
```
- بحث حيّ **مقيّد بقائمة مصادر موثوقة** (whitelist) وداخل `brand.topics` فقط.
- **مالك الـwhitelist (يدعم FR-5):** مصدر القائمة = إعداد `trusted-sources` config = قائمة نطاقات موثوقة عامة (مشتركة لكل المستأجرين) **+** إضافات لكل tenant مشتقّة من محاوره (`brand.topics`). **تُبنى وتُصان ضمن هذه المرحلة (المحرّك)** — لا تابعة لفريق آخر — و**تُحدَّث دون نشر كود** (config لا hardcode).
- جلب الصفحات → استخلاص حقائق مرشّحة، كل واحدة بـ`sourceUrl`.
- **لا مصدر موثوق → `hasFactualClaim=false`**: البوست يكمل كرأي/نبرة بلا ادّعاء فعلي، **وبلا اختلاق مصدر** (NFR-6).
- سقف: عدد عمليات البحث/الجلب لكل بوست محدود (حماية الهامش).

### المرحلة ٢ — كتابة (Claude، خلف ContentProvider)
```ts
interface ContentProvider {
  draft(input: DraftInput): Promise<Draft>
  critique(draft: Draft, rubric: Rubric): Promise<CritiqueResult>
}
interface DraftInput {
  factSet: FactSet
  brand: BrandProfile
  platform: Platform
  contentType: ContentType
  brief?: string
}
interface Draft {
  text: string
  citations: Citation[]   // كل ادّعاء فعلي مقرون بمصدره
  hashtags: string[]
  imageBrief: string      // وصف للصورة لتمريره للمرحلة ٤
}
interface Citation { claim: string; sourceUrl: string }
```
- Claude يكتب بنبرة العلامة، يحوك الحقائق ويقرن كل ادّعاء بمصدره من `FactSet`.
- يحترم **حدود المنصة** من ملف ١٥: العدّ الموزون لـX (عبر `twitter-text`)، ٣٬٠٠٠ للينكدإن، الخطّاف بأول ~١٤٠ حرف للينكدإن، عدد الهاشتاقات (٣–٥ لينكدإن / ١–٢ لـX)، الروابط (X: في ردّ؛ LinkedIn: بلا بطاقة معاينة).
- صياغة مختلفة لكل منصة من نفس الحقائق.

### المرحلة ٣ — نقد ذاتي ونقّح (حلقة الجودة)
```ts
interface Rubric {
  toneMatch: boolean      // مطابقة brand.tone
  sourceIntegrity: boolean// كل ادّعاء فعلي له مصدر صالح
  platformCompliance: boolean // حدود ملف ١٥
  prohibitions: boolean   // لا يخالف brand.prohibitions
  clarity: boolean
}
interface CritiqueResult { score: number; passed: boolean; issues: string[] }
```
- يقيّم المسودّة ضد `Rubric` (مشتقّ من `BrandProfile` + حدود المنصة + الممنوعات).
- `passed=false` → ينقّح ويعيد التقييم. **سقف ٢–٣ دورات**؛ بعدها يمرّر أفضل نسخة مع `issues` للعرض.
- هذه المرحلة هي ضمان "الجودة قبل عين العميل".

### بوابة قرار الصور (gate قبل المرحلة ٤)
- **بوابة قرار رسمية، لا مجرّد ملاحظة:** قبل اعتماد معمارية خط أنابيب الصور (المرحلة ٤) وبدء بنائه، **يُشغَّل اختبار ٢٠ صورة** بنصوص عربية واقعية (من محاور tenant فعليّة، لا نصوص مصطنعة) عبر gpt-image، ويُقاس **معدّل تكسّر النص العربي** بتحقّق بصري.
- **قاعدة القرار:**
  - تكسّر **<١٠٪** → اعتماد `gpt-image` يكتب النص داخل الصورة + طبقة تحقّق بصري (إعادة توليد عند الخطأ) كمسار أساسي.
  - تكسّر **مرتفع (≥١٠٪)** → `overlay` برمجي هو **الأساس** (النص يُركَّب فوق الصورة بخط `kit.font`)، وgpt-image يولّد **الخلفية فقط** بلا نص.
- لا يُعتمد المسار قبل اجتياز هذه البوابة؛ نتيجتها تثبّت قيمة `method` الافتراضية وحدود `attempts` في المرحلة ٤.

### المرحلة ٤ — صورة (gpt-image + تحقّق + overlay احتياط)
```ts
generateImage(brief: string, kit: BrandKit, platform: Platform): Promise<ImageAsset>

interface ImageAsset {
  url: string
  verifiedText: string
  method: 'gpt-image' | 'overlay-fallback'
  attempts: number
}
```
- gpt-image يولّد الصورة بهوية `BrandKit` + النص العربي، بمقاس المنصة (ملف ١٥؛ افتراضي آمن ١٢٠٠×١٢٠٠، المهم في الوسط).
- **تحقّق بصري:** نموذج رؤية يقرأ النص من الصورة ويقارنه بالمقصود → تطابق=اعتماد؛ خطأ/تكسّر=إعادة توليد (سقف ٢–٣) → فشل=إسقاط على **overlay برمجي** بخط `kit.font`.
- **المسار الأساسي (gpt-image vs overlay) محسوم من «بوابة قرار الصور» أعلاه** (اختبار ٢٠ صورة، مهمة بناء #١ من ملف ١٤): البوابة قرار رسمي شرط لبدء بناء هذه المرحلة، لا ملاحظة لاحقة.

### المرحلة ٥ — تجميع وتنسيق
```ts
assemble(draft: Draft, image: ImageAsset, platform: Platform): Post
```
- يدمج النص + الاستشهاد + الصورة، يطبّق إعدادات `platform-limits` (config واحد، لا منثور في الكود).
- يخرج `Post` بحالة `draft → pending_review`.

## خطة الشهر (غير متزامن)
- `generateMonthPlan(brand, count)` = job على **BullMQ** يوزّع `count` خطوط بوست على التقويم السعودي (مناسبات/مواسم) مع **مؤشّر تقدّم** (NFR-2).
- كل بوست يمرّ بالمراحل ١–٥ مستقلاً؛ فشل بوست لا يسقط الخطة (يُعلّم ويُستكمل).
- **تجاوز السقف وسط الخطة:** إن بلغ الاستخدام سقفه أثناء توليد الخطة، البوستات المتبقّية تُعلَّم `skipped_quota` (لا `provider_error`) وتُبلَّغ، **بلا إسقاط الخطة** ولا إعادة محاولة — متوافق مع منطق السقف في المرحلة ٦.

## التعلّم (خفيف عند الإطلاق)
- عند تعديل/اعتماد العميل للبوست: يُلتقط الفرق (الأصل ↔ المعتمد) → يُلخّص → يُضاف لـ`BrandProfile.learnedPreferences`.
- يُحقن في `DraftInput` للتوليد اللاحق، فيتحسّن مع الوقت بلا حلقة مؤتمتة.
- **التعلّم المؤتمت الكامل (من الأداء/التحليلات) = V2** (مؤجّل في الدراسة).

## معالجة الأخطاء (كل مرحلة مستقلة)
| المرحلة | الفشل | السلوك |
|---|---|---|
| بحث | لا مصدر موثوق | بوست رأي/نبرة بلا ادّعاء، بلا اختلاق مصدر |
| كتابة | خطأ مزوّد (`provider_error`) | إعادة محاولة → طابور؛ تسجيل في UsageRecord |
| أي مرحلة | **تجاوز السقف وسط الخطة** (`skipped_quota`) | **يُميَّز عن فشل المزوّد:** البوست يُعلَّم `skipped_quota` ويُبلَّغ، **بلا إسقاط بقية الخطة** ولا إعادة محاولة (السقف، لا عطل)؛ متوافق مع المرحلة ٦ |
| نقد | لا يعدّي بعد ٣ دورات | يمرّر أفضل نسخة + issues ظاهرة للعميل |
| صورة | تكسّر بعد ٣ محاولات | overlay احتياط؛ وإن فشل → نص فقط بعلامة |
| تجميع | تجاوز حدود المنصة | يُعاد للمرحلة ٢ بقيد أشدّ |

> **تمييز جوهري:** `skipped_quota` (وصول سقف الاستخدام وسط تنفيذ خطة الشهر) ≠ `provider_error` (عطل مزوّد). الأول قرار حدّ متوقّع: يُعلَّم البوست ويُبلَّغ ويُكمَل الباقي؛ الثاني خطأ تشغيلي يُعاد محاولته ويُسجّل في `UsageRecord`. خلطهما يسبّب إعادة محاولات لا طائل منها على بوستات تجاوزت السقف.

## الواجهات الخارجية المستهلَكة
- `ContentProvider` → Claude (نص).
- `ImageProvider` → OpenAI gpt-image + `VisionVerifier` (تحقّق) + `OverlayRenderer` (احتياط، Satori/Sharp).
- `SearchProvider` → بحث حيّ مقيّد بمصادر موثوقة. **ملكية التنفيذ:** المنفّذ الحقيقي لـ`SearchProvider` (البحث الحيّ المقيّد بالـ`trusted-sources` whitelist + جلب الصفحات + استخلاص الحقائق) **يُبنى ضمن مرحلة المحرّك (١)** — ليس seam مستهلَك من فريق/مرحلة أخرى، بل مكوّن أساسي تملكه هذه المرحلة وتختبره.
- `platform-limits` config (ملف ١٥) + `twitter-text` للعدّ.

## ما هو خارج هذا الـspec
الواجهة (frontend)، الفوترة (Moyasar)، الـAuth، التقويم UI، النشر اليدوي UI — لها specs/مراحل منفصلة. هذا الملف يغطّي **المحرّك فقط**.

**داخل النطاق صراحةً (لئلا يُظنّ خارجه):** تنفيذ `SearchProvider` الفعلي + إعداد `trusted-sources` whitelist وصيانته — كلاهما يُبنى ويُملَك ضمن مرحلة المحرّك، لا يُورَّث لفريق آخر.
